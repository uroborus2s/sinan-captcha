## 项目背景

当前仓库已经不是“只有种子文档的需求孵化项目”。仓库内已经存在 Go 生成器、Python 训练 CLI、自主训练控制器骨架，以及统一求解与 bundle 的部分实现。

此前正式文档的主叙事仍偏向“零基础训练工程”，这会把后续设计、发布和交付持续引向错误方向。用户在 2026-04-05 明确确认：

- 最终业务产物不是分散的训练脚本和权重目录，而是 1 个统一验证码求解包/库。
- `group2` 的业务语义是：给定背景图和缺口图，输出缺口图在背景图上的目标中心点。
- `group1` 的业务语义是：给定查询图和背景图，按查询图中的图标顺序，在背景图上输出对应图标的中心点序列。
- Go 生成器、Windows 训练 CLI 和自主训练 CLI 都是为了生产这个最终求解包/库的支撑能力，而不是代替它的最终产品。

## 一级产品定义

- 统一求解包/库：
  - 对外只暴露 1 套稳定业务语义。
  - 对外交付 1 个可复制的模型 bundle 目录。
  - 调用方不需要理解 `group1` 的多个内部部件，也不需要自己编排训练目录。
- `group2` 求解能力：
  - 输入背景图和缺口图。
  - 输出背景图坐标系中的目标中心点；必要时附带目标框、偏移量等辅助字段。
- `group1` 求解能力：
  - 输入查询图和场景图。
  - 输出按查询顺序排列的中心点序列；必要时附带顺序状态、歧义状态和匹配证据。
- 模型生产工具链：
  - `sinan-generator` 负责受控生成、素材管理、真值校验和训练数据导出。
  - `sinan` 负责训练目录、训练、测试、评估、发布和自主训练入口。
  - 自主训练负责用受限 agent 和本地控制器循环提升最终 bundle 质量。

## 当前已确认的核心判断

- 项目一级产品应是统一验证码求解包/库，而不是训练工程本身。
- 所有业务坐标都以目标中心点定义，不能在不同文档里混用左上角、拖动距离和中心点语义。
- `group1` 与 `group2` 的业务输入输出不同，但对外交付应统一到 1 套入口与 1 套 bundle 口径。
- 训练、样本生成和自主训练仍然是必要能力，但它们属于模型生产线，不应继续占据公开文档的一级产品位置。

  - 当前仓库已具备哪些稳定能力；
