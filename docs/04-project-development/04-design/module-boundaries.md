# 模块边界基线

- 项目名称：sinan-captcha
- 当前阶段：DESIGN
- 当前技术栈：Python, PyTorch, Ultralytics YOLO（group1）, Windows, CUDA, X-AnyLabeling/CVAT, OpenCode, Optuna, SQLite

## 边界设计原则

1. 先围绕“数据闭环”划分模块，而不是围绕文件类型随意堆脚本。
2. 把“生成/导出”“标注”“训练”“评估”拆开，避免一个脚本承担所有职责。
3. 把第一专项和第二专项共享的能力抽到公共模块，任务特有逻辑单独隔离。
4. 内部生成器优先是数据模块，不等于对外业务验证码平台。
5. 生成器控制层拥有训练契约和 `gold` 定义权，backend 只能作为可插拔能力源。

## 模块清单

### MOD-001 生成器适配层

- 职责：
  - 连接自有验证码生成逻辑或内部生成器 backend
  - 统一导出图形点选和滑块两类样本的图片与真值标签
  - 维护 backend 选择、随机种子、批次元数据和阻断规则
  - 维护内置 preset、workspace 固定命名覆盖和像素级视觉增强参数
  - 产出通过真值校验的原始样本批次
- 不负责：
  - 模型训练
  - 人工审核 UI
  - 评估报告
- 输入：
  - 生成参数、模板、背景、图形资源、backend 配置
  - preset 与 workspace `presets/` 覆盖文件
- 输出：
  - `raw` 目录图片
  - JSONL 标签
- 依赖：
  - 自有生成端或开源底座（如 `go-captcha`），但只能通过适配层接入
- 禁止耦合：
  - 不直接绕过数据契约层写入任务专属训练数据集
  - 不允许绕过控制层直接把第三方接口返回值当作 `gold`
  - 不允许 backend 自行定义会改变标签几何的视觉增强参数

### MOD-002 数据契约与版本管理

- 职责：
  - 定义 JSONL schema
  - 维护数据版本、切分规则和类别表
  - 维护 `gold` / `auto` / `reviewed` 状态
- 不负责：
  - 生成图片
  - 训练模型
- 拥有数据：
  - 类别表
  - 数据集版本元数据
  - 批次统计信息
- 禁止耦合：
  - 不直接调用训练框架

### MOD-003 自动标注流水线

- 职责：
  - 第二组规则法预标注
  - 第一组暖启动模型批量预标注
  - 输出 `auto` 标签
- 不负责：
  - 最终人工审核
  - 训练集切分的事实判定
- 输入：
  - `raw` 样本
  - 第一组暖启动权重
- 输出：
  - `interim` 标签
- 错误策略：
  - 预标注质量低于阈值时，阻断进入 `reviewed`

### MOD-004 标注复核与转换层

- 职责：
  - 将 `auto` 或 `gold` 标签导入 X-AnyLabeling / CVAT
  - 汇总抽检结果
  - 生成 `reviewed` 标签
  - 把 `reviewed` / `gold` 数据转换成训练 CLI 可直接消费的任务专属数据集
- 不负责：
  - 生成图片
  - 模型训练
- 输入：
  - `gold` / `auto` 标签
- 输出：
  - `reviewed` 标签
  - `group1 yolo/images/*`
  - `group1 yolo/labels/*`
  - `group1 dataset.yaml`
  - `group2 master/*`
  - `group2 tile/*`
  - `group2 splits/*.jsonl`
  - `group2 dataset.json`

### MOD-005 第一专项训练模块

- 职责：
  - 训练第一组多类别检测模型
  - 管理第一专项超参数、权重和日志
  - 输出检测结果
- 不负责：
  - 第二专项训练
  - 第一组顺序映射之外的检索模型
- 输入：
  - 第一组 `reviewed` 数据
  - 预训练权重
- 输出：
  - 第一专项模型权重
  - 训练日志
  - 中间评估结果

### MOD-006 第二专项训练模块

- 职责：
  - 训练第二组滑块缺口定位模型
  - 管理第二专项超参数、权重和日志
- 不负责：
  - 规则法本身的长期维护策略决策
  - 第一专项训练
- 输入：
  - 第二组 paired dataset
  - 初始化方式或历史检查点
- 输出：
  - 第二专项模型权重
  - 训练日志
  - 中间评估结果

### MOD-007 推理后处理与任务映射

- 职责：
  - 第一组把检测结果映射成顺序点击点
  - 第二组把检测框换算成中心点和偏移量
  - 统一输出推理契约
- 不负责：
  - 模型训练
  - 数据标注
- 输入：
  - 模型预测结果
  - 查询顺序或目标定义
- 输出：
  - 第一组点击点列表
  - 第二组点位、框与偏移量

### MOD-008 评估与报告模块

- 职责：
  - 计算第一组和第二组指标
  - 归档失败样本
  - 产出模型版本报告
- 不负责：
  - 数据生成
  - 标注审核
- 输出：
  - 指标报表
  - 失败样本清单
  - 版本对比摘要

### MOD-009 自主训练控制器

- 职责：
  - 维护 study 生命周期、trial 编号、停止规则和恢复逻辑
  - 调用生成、训练、测试、评估等既有 CLI
  - 校验 AI 决策 JSON 并执行下一轮动作
- 不负责：
  - 直接生成模型判断结论
  - 直接提供人类可读长报告之外的解释
- 输入：
  - study 配置
  - runner 执行结果
  - AI 决策 JSON
- 输出：
  - `study.json`
  - `trial_history.jsonl`
  - 各 trial 工件目录

### MOD-010 agent skills 与命令接入层

- 职责：
  - 通过 `opencode` project-local commands 和 skills 承担结果摘要、判断、数据规划和归档建议
  - 约束 AI 的动作空间和返回 JSON 契约
- 不负责：
  - 直接持久化 study 状态
  - 直接运行训练命令
- 输入：
  - `test.json`
  - `evaluate.json`
  - 最近 N 轮摘要
- 输出：
  - `decision.json`
  - 数据规划建议
  - study 摘要建议

### MOD-011 优化与策略模块

- 职责：
  - 维护 `group1`/`group2` 的目标函数、停止规则和参数搜索空间
  - 驱动 `Optuna` 选择下一轮参数
- 不负责：
  - 直接解释业务结论
  - 直接读写原始日志
- 输入：
  - 当前最优结果
  - 最近 trial 指标
  - AI 决策方向
- 输出：
  - 下一轮参数候选
  - pruning/plateau 结论

## 禁止耦合关系

- 第一专项训练模块不得直接读取第二专项私有标签逻辑
- 自动标注模块不得绕过数据契约层直接写训练目录
- 评估模块不得直接篡改 `reviewed` 标签
- 生成器适配层不得直接依赖训练框架
- `opencode` commands/skills 不得直接接管训练 shell
- 控制器不得把 AI 输出当作无需校验的命令执行

## 错误与降级策略

- 生成端不可用时：
  - 降级为小规模种子集 + 暖启动模型路线
- 生成端返回值无法证明真值正确时：
  - 阻断写入 `gold`
  - 回退到修正 backend 适配或显式改为 `auto` 路线
- 自动标注准确率不足时：
  - 阻断进入 `reviewed`
- 第一专项类别体系不稳定时：
  - 发起设计变更，不在实现中临时热修
- 第二专项规则法失效时：
  - 保留规则法为对照工具，但不替代模型训练目标
- AI 决策 JSON 非法或与当前状态冲突时：
  - 回退到规则模式
  - 当前 trial 仍需完整落盘
