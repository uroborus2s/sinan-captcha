# 头脑风暴记录

- 初始时间：2026-04-01
- 最近更新：2026-04-11
- 阶段：BRAINSTORM -> REQUIREMENTS -> 需求基线重构
- 负责人：Codex

## 项目背景

项目最初从一份图形验证码训练手册启动，最早的决策重点是：

- 这不是 OCR 问题，而是图形定位问题。
- `group1` 与 `group2` 应拆成两条专项任务，不做一个混合大模型。
- 自动标注优先于大规模人工标注。
- Windows + NVIDIA 单机的新手可执行性是第一优先级。

随着仓库进入实现阶段，Go 生成器、Python 训练 CLI、自主训练控制器骨架和统一求解骨架逐步出现，但公开文档仍主要把项目描述成“训练工程”。2026-04-05，用户进一步明确最终业务要求：项目真正要交付的是一个本地可调用的统一验证码求解包/库。

## 最新明确的用户意图

- 最终调用方只希望面对 1 个统一求解包/库，而不是若干训练命令和分散权重目录。
- `group2` 的最终业务值是背景图上的目标中心点。
- `group1` 的最终业务值是按查询顺序排列的中心点序列。
- 生成器、训练 CLI 和自主训练 CLI 都必须保留，但它们是为最终求解包持续生产模型的内部能力。
- 部署文档必须覆盖训练机、素材、生成器和自主训练，而不只是“怎么开始训练”。

## 方案比较

### 方案 A：继续维持训练工程优先

- 思路：继续把仓库定义为“本地双 CLI 训练工程”，统一求解包只作为后续增强项。
- 优点：与当前最完整的实现主线一致，改动最少。
- 缺点：会继续把对外交付和内部生产线混成一个产品；后续设计、打包和部署都会被训练视角牵着走。
- 结论：不采用。

### 方案 B：把求解包和模型生产工具链拆成两个仓库

- 思路：把 solver 单独建仓，当前仓库只保留 generator/train/auto-train。
- 优点：产品边界非常清晰。
- 缺点：当前实现、文档和版本关系都已经缠在同一仓库，立即拆仓成本高，且会增加同步复杂度。
- 结论：当前不采用。

### 方案 C：单仓库保留两层产品面，但把求解包提升为一级产品

- 思路：保持当前单仓结构，明确“统一求解包/库”是一级产品；Go 生成器、训练 CLI 和自主训练 CLI 是二级支撑能力。
- 优点：既能保留现有实现基础，又能修正需求、设计、发布和公开文档的方向；后续代码收口路径也更清晰。
- 缺点：文档必须严格区分“当前稳定实现”和“最终目标交付”，否则容易再次混乱。
- 结论：采用。

## 最终决定

- 项目一级产品正式收口为本地统一验证码求解包/库。
- `group1` 和 `group2` 的业务语义都以中心点坐标为最终结果。
- Go 生成器、Windows 训练 CLI、自主训练 CLI 保留为模型生产工具链。
- 正式文档必须把“当前已实现训练主链路”与“最终 solver 交付主线”分层表达。
- 后续设计和实现应优先补齐统一求解入口、bundle 打包和部署文档，而不是继续把这些能力放在附属位置。

## 2026-04-11 新增收口：`group1` 从闭集类名检测改为实例匹配求解

在 2026-04-10 到 2026-04-11 的连续问题澄清后，`group1` 的关键矛盾已经进一步暴露：

- 真实 query 图中持续出现训练类表之外的新图标。
- 人工审核和商业试卷制作长期被类名表绑住，审核成本高且容易误判。
- 旧路线的核心假设是“模型只需要判断这是不是某个已知类”，但真实业务更接近“query 里的这个图标，在 scene 里是哪一个实例”。

基于这个事实，本轮新增比较了 3 种 `group1` 路线：

### 方案 D：继续沿用闭集类名路线

- 思路：继续维护 `query parser + scene detector + ordered_class_match_v1`。
- 优点：和当前代码最接近，短期迁移成本最低。
- 缺点：
  - 新图标持续出现时，模型主合同会不断失效；
  - 商业试卷和人工审核长期依赖类名表；
  - `class_id` 只能回答“是不是这一类”，不能稳定回答“是不是同一个图标实例”。
- 结论：不采用。

### 方案 E：直接做端到端图像条件检测大模型

- 思路：把 query 图和 scene 图一起送进单一大模型，直接输出有序点击点。
- 优点：概念上最简洁，对调用方最像“一个模型解决问题”。
- 缺点：
  - 当前仓库训练、导出、部署和自动训练主线都要整体重写；
  - 对单机 Windows + NVIDIA 的首版可执行性不友好；
  - 调参、定位问题和商业验收都不够可解释。
- 结论：当前不采用。

### 方案 F：实例匹配式求解

- 思路：
  - query 先切出图标项；
  - scene 先做类无关图标候选检测；
  - 再用图标向量匹配和全局分配输出有序中心点。
- 优点：
  - 不再把闭集类名当 solver 主合同；
  - 新图标只要进入素材和真值体系，就可以纳入训练与评估；
  - 人工审核和商业试卷可以退化成“只标顺序和框”，不再要求维护庞大类名表。
- 缺点：
  - 需要重构生成器、数据契约、训练链和自动训练策略；
  - 必须显式淘汰当前旧 `group1` 路线，不能长期双轨并存。
- 结论：采用。

## 2026-04-11 最终决定补充

- `group1` 正式设计主线改为：
  - `query splitter`
  - `scene proposal detector`
  - `icon embedder`
  - `global assignment matcher`
- `group1` 素材库仍然保留语义分类，但分类只作为素材治理辅助，不再作为 solver 主合同。
- `group1` 的训练主事实源改为 `asset_id / template_id / variant_id` 这类实例身份，而不是 `class_id`。
- 商业试卷、人工审核和预标注要改成“顺序 + 框”优先，而不是“类名 + 框”优先。
- 这次重构不接受长期兼容旧方案：
  - 文档切换后，后续实施必须删除被舍弃方案的正式代码、测试、CLI 入口和用户文档，保持单一正式主线。

## 约束条件

- 不回退到 OCR 或图像分类的任务定义。
- 不把自由 shell 代理作为自主训练的默认形态。
- 不在文档里宣称尚未正式接入发布主线的能力已经完成交付。
- 不放弃 Windows + NVIDIA 单机和零基础可执行性约束。
