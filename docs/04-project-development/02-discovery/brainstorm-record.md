## 项目背景

用户希望在一个非空目录里启动 `sinan-captcha`（司南）项目。目录中已有一份 `graphical_captcha_training_guide.md`，其中已经把问题定义为两类图形验证码定位任务，而不是 OCR。

用户的关键要求是：

- 先按软件工厂规则处理目录状态
- 进入需求阶段，不跳到实现
- 给零基础用户形成真正能落地的训练方案
- 重点解决 Windows + NVIDIA 环境搭建、样本获取、自动标注和训练验收

## 约束条件

- 当前目录非空，不能直接走空仓初始化
- 只有一台 Windows + NVIDIA 电脑可用
- 用户有基础 IT 能力，但没有模型训练经验
- 不希望大量人工标注

## 方案比较

### 方案 A：沿用 OCR/分类训练工具路线

- 思路：直接参考 `dddd_trainer` 或 `captcha_trainer` 的默认任务模式，按 OCR/分类项目推进。
- 优点：对新手友好，现成资料多。
- 缺点：任务定义错误。当前问题输出的是坐标，不是字符串或单张图片类别。
- 结论：不采用。

### 方案 B：一个统一模型覆盖两类验证码

- 思路：把第一组和第二组样本混在一起，训练一个统一检测或分割模型。
- 优点：表面上模型数量少。
- 缺点：输入输出不统一、标注格式不统一、训练互相干扰，新手更难调通。
- 结论：不采用。

### 方案 C：两个专项模型 + 自动标注优先

- 思路：第一组用多类别检测模型，第二组先按单目标定位思路建模；优先从生成端导出标签，没有金标签时再用规则法和标注工具做预标注。
- 优点：任务清晰、数据结构清晰、对新手最稳，也符合现有训练手册判断。
- 缺点：需要提前整理数据契约和类别表。
- 结论：采用。

## 最终决定

- 项目正式进入需求阶段。
- 第一专项模型按多类别检测定义，不把 OCR 或分类当主线。
- 第二专项模型最初按单类别目标检测定义，同时允许规则法承担预标注和基线评估角色。
- 自动标注路线优先级高于人工逐张标注。
- 新手可执行性是首要设计约束。

## 需要后续继续明确的事项

- 第一组图标类别是否足够稳定
- 第二组目标是否会扩展为多形状
- 生成端能否直接导出坐标
- 目标机器显存规模与磁盘空间上限
