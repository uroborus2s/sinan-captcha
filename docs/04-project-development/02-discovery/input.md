# 项目输入

- 项目名称：sinan-captcha
- 中文名：司南
- 当前阶段：IMPLEMENTATION（本轮执行需求与文档校正）
- 核心创意：构建一个本地可调用的统一验证码求解包/库，并在同一仓库内提供受控样本生成、Windows GPU 训练和自主训练能力，持续生产可交付的模型 bundle
- 用户条件：已有一台 Windows 电脑和 NVIDIA 显卡；有一定 IT 基础；没有模型训练经验
- 本轮目标：把正式需求和公开文档从“训练工程优先”改写为“最终求解包/库优先”，同时保留完整的模型生产工具链要求

## 2026-04-05 最终业务澄清

### 最终产物

1. `group2` 拖动式验证码求解能力：
   - 输入背景图和缺口图。
   - 输出缺口图在背景图上的目标中心点。
2. `group1` 点选式验证码求解能力：
   - 输入查询图和背景图。
   - 按查询图中的图标顺序，在背景图上输出对应图标的中心点序列。
3. 最终对外交付形式：
   - 1 个统一求解包/库。
   - 1 个可复制的 bundle 目录。
   - 1 套统一业务语义和请求/响应合同。

### 项目分解

1. Go 生成器 CLI：
   - 根据背景素材和图标素材生成 `group1` 与 `group2` 样本。
   - 输出图片、标注、真值和 QA 结果。
2. Windows 训练 CLI：
   - 在 Windows + NVIDIA 训练机上消费数据集，训练、测试、评估并产出可交付模型。
3. 自主训练 CLI：
   - 调整生成器参数、自动生成样本、训练、分析结果、判断丢弃或调参重试，循环到满足业务要求或触发停止条件。
4. 最终求解包发布：
   - 将训练好的模型、配置和合同包装为方便调用的库或等价本地求解入口。

## 2026-04-11 增量业务约束（solver 输入兼容）

- 调用方不应被迫先把图片落地为本地路径后再调用 solver。
- 正式 `sinanz` 入口必须支持统一图片输入适配层，至少覆盖：
  - 本地路径（`str | Path`）
  - 二进制图片（`bytes`）
  - base64 文本（含 `data:` URI）
  - 网络图片路径（`http://`、`https://`）
- “支持所有图片格式”在工程口径上定义为：支持所有被运行时解码器（Pillow）可稳定解码的常见图片格式，并在失败时返回结构化错误，而不是静默失败。
- 网络图片输入必须受安全边界约束：协议白名单、大小上限、超时、重定向限制与可审计日志。

## 已知约束

- 当前仓库已经存在代码与文档，本轮是需求校正，不是从零启动。
- 正式文档必须区分“当前稳定实现”和“最终目标交付”，不能把未完成能力写成既成事实。
- `group1` 与 `group2` 的业务坐标都以中心点定义。
- 自主训练仍要保持“确定性执行归 Python 控制器、解释与决策归受限 agent”的边界，不能退化为自由发挥式 shell 代理。
- 样本来源必须来自自有或已授权环境，不得引入来源不明的抓取数据。

## 外部参考的吸收结论

- `dddd_trainer` 和 `captcha_trainer` 更适合借鉴“新手友好的 Windows GPU 训练体验”，不适合作为当前业务语义和最终交付方式的直接模板。
- 本项目的正式一级产品仍然不是 OCR 工具、分类器或通用 AI 训练框架，而是围绕两类验证码业务语义构建的求解包/库。
- 模型训练框架、标注工具和自动化训练能力都属于求解包生产线，而不是最终用户直接消费的主产品定义。
