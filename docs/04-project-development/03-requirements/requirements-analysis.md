# 需求分析文档

- 文档状态：草稿
- 当前阶段：REQUIREMENTS
- 最近更新：2026-04-02
- 负责人：Codex
- 备注：本分析聚焦零基础用户、Windows + NVIDIA 单机训练、自动标注优先，以及“两专项模型”而非统一 OCR 模型；其中生成器按“受控集成 + 可插拔 backend”收口。

## 分析原则

- 先确认任务定义是否正确，再讨论模型细节；任务定义错误会直接导致训练方向失效。
- 优先选择对新手最稳、自动化程度最高、反馈周期最短的路径。
- 需求必须服务于后续设计，不在本阶段偷跑实现。
- 把“自动标注”和“数据治理”视为首版成功的前置条件，而不是补充项。

## 逐条分析

### REQ-001

- 业务价值：环境稳定是零基础用户能否真正开始训练的首要门槛，没有可复现环境，后续所有数据和模型工作都会卡住。
- 优先级理由：P0。它阻塞样本处理、预标注、训练和验收全部后续动作。
- 依赖关系：依赖目标机器具备 Windows + NVIDIA 显卡；依赖 PyTorch 与显卡驱动兼容；依赖项目确定统一 Python 管理方式。
- 风险点：Windows 上最容易出现驱动、CUDA、PyTorch 版本不兼容；新手容易把依赖装进系统 Python 导致环境污染。
- 影响角色/系统：主要影响零基础训练操作者和项目维护者，也影响后续文档维护和问题排查效率。
- 待澄清问题：用户显卡型号和显存大小尚未提供；是否允许在公司环境安装额外驱动和工具尚未确认。
- 建议拆分：拆为“环境安装标准”“GPU 自检标准”“冒烟训练标准”三个子部分。
- 建议测试点：`nvidia-smi` 输出存在；`torch.cuda.is_available()` 为 `True`；YOLO 训练命令能够跑完 1 个 epoch 的冒烟集。

### REQ-002

- 业务价值：自动导出金标签是本项目降低人工成本的关键，直接决定数据质量上限和交付速度。
- 优先级理由：P0。没有稳定数据来源，自动标注只能退化为脆弱的规则猜测。
- 依赖关系：依赖验证码生成端、渲染日志或其他可审计数据源；依赖字段设计先于数据采集固定。
- 风险点：若无法接触生成端，第一组顺序标签和干扰项标签、第二组滑块缺口真值都很难完整恢复；数据切分不当会引入泄漏；若 backend 返回的数据不能直接映射到训练契约，后续会产生大规模返工。
- 影响角色/系统：影响生成服务维护者、项目维护者、数据处理脚本和训练评估流程。
- 待澄清问题：当前是否掌握生成端源码或渲染参数；是否已有现成样本库存；是否存在数据合规限制。
- 建议拆分：拆为“第一组导出契约”“第二组滑块导出契约”“gold 真值校验规则”“数据切分与审计规则”。
- 建议测试点：导出字段完整性、样本去重、训练/验证/测试集合互斥、来源标识可追溯、真值一致性与重放校验。

### REQ-003

- 业务价值：自动标注把项目从“需要标注团队”降到“个人可推进”，是零基础落地方案的核心。
- 优先级理由：P0。用户明确要求“自动标记而非人工”，且样本量达到几千到几万级，纯人工不可持续。
- 依赖关系：依赖 REQ-002 提供原始图片或金标签；依赖标注工具选型；依赖规则法/预标注脚本可运行。
- 风险点：自动标注精度不足会污染训练集；第一组查询图与场景图匹配逻辑不稳时，伪标签可能系统性偏差；如果把生成器直出样本和自动标注样本混用而不区分状态，会把“应当 100% 正确的 gold”降级成普通候选标签。
- 影响角色/系统：影响零基础训练操作者、项目维护者、标注工具和后续训练结果。
- 待澄清问题：用户更偏好桌面工具还是 Web 标注工具；是否能接受“自动预标 + 人工抽检”而不是“百分百零人工”。
- 建议拆分：拆为“金标签直出”“真值一致性校验”“规则法预标”“工具复核与状态流转”“抽检标准”。
- 建议测试点：预标注覆盖率、抽检准确率、标签状态机、导入导出回归、gold 阻断机制。

### REQ-004

- 业务价值：固定数据结构和版本策略，才能支持长期迭代、多人协作和结果复现。
- 优先级理由：P0。没有数据治理，模型再好也无法稳定维护。
- 依赖关系：依赖 REQ-002、REQ-003 的样本和标签事实源；依赖训练框架采用明确的数据格式。
- 风险点：不同任务混放会造成标签混乱；测试集不冻结会导致指标失真；第一组类别表可能失控增长。
- 影响角色/系统：影响数据处理脚本、训练脚本、评估报告和后续模型版本比较。
- 待澄清问题：第一组预计有多少稳定图标类别；未来是否会引入新的验证码类型。
- 建议拆分：拆为“目录结构”“标签主事实源”“训练转换物”“版本冻结与统计”。
- 建议测试点：目录结构校验、schema 校验、类别表一致性、测试集冻结校验。

### REQ-005

- 业务价值：第一专项模型是项目最具难度和价值的核心交付之一，直接对应多图标点击验证码。
- 优先级理由：P0。用户的核心创意之一就是“训练 1 个多类别检测模型”。
- 依赖关系：依赖第一组数据集、类别表、查询顺序字段、自动标注质量和训练环境。
- 风险点：如果同类图标在同一图中多次出现，单纯按类别映射会歧义；如果类别体系不稳定，模型维护成本会上升。
- 影响角色/系统：影响训练流程、推理输出定义、评估指标和后续接口设计。
- 待澄清问题：第一组图标类别是否固定且数量可控；查询图是否总能明确顺序；是否允许未来引入匹配分支。
- 建议拆分：拆为“多类别检测训练”“查询顺序映射”“重复类别消歧”“失败样本回灌”。
- 建议测试点：点位命中率、整组顺序命中率、重复类别场景、未知类别样本处理。

### REQ-006

- 业务价值：第二专项模型难度较低，适合先形成稳定闭环，也能为新手建立信心和验证整个数据训练链路。
- 优先级理由：P0。它是第二个正式专项模型，同时还能承担规则法预标和模型对照作用。
- 依赖关系：依赖第二组数据集、规则法基线、自动标注质量和环境可用性。
- 风险点：若第二组未来演化成多缺口、多目标或弱轮廓滑块场景，当前定位方案将失效；规则法如果过弱，预标注价值下降；如果生成器不能稳定导出滑块缺口真值，第二专项训练会直接失去可靠事实源。
- 影响角色/系统：影响训练脚本、自动标注、评估脚本和失败样本分析。
- 待澄清问题：第二组是否始终是单缺口滑块；是否需要预测完整轨迹还是只预测缺口位置与偏移；背景复杂度上限未知。
- 建议拆分：拆为“规则法基线”“滑块缺口定位训练”“偏移估计”“规则法与模型对照”。
- 建议测试点：点位误差、IoU、偏移误差、复杂背景鲁棒性、规则法与模型输出差异分析。

### REQ-007

- 业务价值：没有真正可执行的操作手册，项目仍然只对熟手友好，无法满足“小白可落地”的目标。
- 优先级理由：P0。用户明确要求“毫无模型训练经验和一无所知的人能执行的操作步骤”。
- 依赖关系：依赖前六条需求先把术语、路径、目录和动作定义清楚。
- 风险点：文档如果只写概念不写成功标志和失败处理，新手仍会被卡死；文档与真实目录不一致会快速失效。
- 影响角色/系统：影响零基础训练操作者、项目维护者和后续交接效率。
- 待澄清问题：最终文档是更偏“命令行教程”还是“图文手册”；是否需要配套短视频或截图。
- 建议拆分：拆为“准备清单”“操作步骤”“常见错误”“验收方法”。
- 建议测试点：新人走查、卡点统计、常见故障分类准确率、文档路径一致性。

### REQ-008

- 业务价值：把训练活动工程化，避免模型文件散落、指标不可比和失败样本丢失。
- 优先级理由：P1。它不阻塞第一轮训练启动，但决定项目能否持续迭代。
- 依赖关系：依赖训练流程、冻结测试集和统一的模型版本命名规则。
- 风险点：如果测试集随着训练一起变化，版本比较失真；如果失败样本没有归档，模型无法定向优化。
- 影响角色/系统：影响项目维护者、训练报告、模型存档和后续部署准备。
- 待澄清问题：最终模型是否需要导出 ONNX；是否要为未来部署预留统一文件命名与目录结构。
- 建议拆分：拆为“版本化产物清单”“评估报告模板”“失败样本回灌规则”。
- 建议测试点：产物完整性、模型版本可追溯性、失败样本去重、评估报告一致性。

## 总体判断

- 当前最稳的主线是“环境先通 -> 受控生成器导出与真值校验 -> 自动标注/抽检 -> 第二组基线 -> 两专项模型训练 -> 验收回灌”。
- 外部参考仓库更适合作为“新手训练体验”的借鉴，而不是直接照搬模型任务定义。
- 本项目最大的成功关键不在于选哪一个网络，而在于能否拿到可审计的金标签、把 backend 能力约束在自有契约后面，并对 gold 真值建立不可绕过的阻断机制。
