# 需求分析文档

- 文档状态：已重构
- 项目当前阶段：IMPLEMENTATION
- 本轮动作：需求基线重构
- 最近更新：2026-04-12
- 负责人：Codex
- 备注：本分析围绕“统一验证码求解包/库”这一一级产品重新组织需求，并将生成器、训练 CLI 和自主训练 CLI 明确为支撑能力

## 分析原则

- 先锁定最终业务产物，再安排生产它的工具链。
- 需求优先级应围绕“调用方最终拿到什么”排序，而不是围绕“当前哪段代码更完整”排序。
- 文档必须忠实反映当前事实：训练与评估主链路较完整，统一求解和打包主线仍在收口。
- 需求分析既要说明业务价值，也要明确对当前项目偏差的矫正方向。

## 逐条分析

### REQ-001

- 业务价值：它直接决定项目最终卖出去、交付出去的是什么。如果没有统一求解包和单一调用入口，项目最终仍然只是训练工程。
- 优先级理由：P0。它是整个产品重定位的核心。
- 依赖关系：依赖 `group1`、`group2` 的模型产物、bundle 目录规范和统一响应合同。
- 风险点：若统一入口只停留在设计文档里，不进入发布主线，公开定位会再次漂移；若对外合同泄露内部子模型结构，调用方负担仍然过高。
- 影响角色/系统：直接影响调用方、发布流程、统一求解服务设计和文档结构。
- 建议测试点：统一入口成功调用、冲突请求阻断、统一错误对象回归。

### REQ-002

- 业务价值：`group2` 是最终求解包的一个正式业务能力，输出中心点而不是训练指标本身。
- 优先级理由：P0。没有它，就无法完成拖动式验证码场景的业务交付。
- 依赖关系：依赖 `group2` 数据集、训练 CLI、评估指标和 bundle 封装。
- 风险点：若把偏移量、起始框等技术字段当成主结果，会模糊最终业务语义；若中心点语义在不同文档里不一致，会导致调用方接入错误。
- 影响角色/系统：影响统一响应合同、评估基线、部署说明和后续接口适配。
- 建议测试点：中心点响应、附加调试字段、错误输入处理。

### REQ-003

- 业务价值：`group1` 体现了项目最复杂、最有价值的真实业务输入，不能被简化成“检测几个框”或“分类几个类名”。
- 优先级理由：P0。没有有序中心点序列，就不构成点选式验证码求解能力。
- 依赖关系：依赖 `query splitter`、`scene proposal detector`、`icon embedder`、`matcher`、统一输出合同和可解释状态。
- 风险点：
  - query 切分错误会直接导致顺序错位；
  - scene 候选召回不足会导致漏点；
  - 相似图标之间的向量混淆和全局分配错误是新的主风险；
  - 如果对外只返回无序候选框或继续要求闭集类名，业务价值都不成立。
- 影响角色/系统：影响模型设计、推理逻辑、评估指标和 bundle 结构。
- 建议测试点：顺序命中、相似图标区分、歧义拒判、失败状态而非静默错误。

### REQ-004

- 业务价值：bundle 和交付包把模型从训练目录里“带出来”，没有这一步就谈不上真正交付。
- 优先级理由：P0。它是统一求解包的发布实体。
- 依赖关系：依赖模型版本冻结、manifest 设计、release 规则和部署文档。
- 风险点：若 bundle 依赖绝对路径或训练目录结构，交付后会立即失效；若交付包缺少说明和版本映射，维护成本极高。
- 影响角色/系统：影响交付使用者、发布流程、部署和回滚。
- 建议测试点：跨路径加载、缺失文件阻断、交付包完整性。

### REQ-005

- 业务价值：受控生成器是最终求解包质量的上游事实源，直接决定数据质量上限。
- 优先级理由：P0。没有稳定 gold 样本，后续训练和自动化优化都建立在脆弱基础上。
- 依赖关系：依赖素材规范、工作区规范、truth-preserving preset 和 QA 规则。
- 风险点：
  - 如果仍让 `group1` 只围绕类名导出数据，真实新图标会持续脱靶；
  - 如果 `asset_id`、来源和实例身份不稳定，embedding 训练会不可比较；
  - 如果允许坏样本、坏素材或语义漂移 preset 混入训练集，最终 bundle 指标会虚高且不可复现。
- 影响角色/系统：影响生成服务维护者、数据治理、训练效果和自主训练。
- 建议测试点：真值校验、坏素材阻断、`asset_id` 稳定性、导出契约稳定性。

### REQ-006

- 业务价值：训练 CLI 是把样本变成最终模型的确定性生产工具。
- 优先级理由：P0。没有稳定训练闭环，统一求解包无法持续产出新版本。
- 依赖关系：依赖 Windows GPU 环境、数据集契约、评估与测试规则。
- 风险点：
  - 如果 `group1` 评估仍停留在旧闭集检测指标，训练目标会和真实业务门脱节；
  - proposal detector、embedder 和 matcher 之间若没有统一验证链，试验结果会失真；
  - 训练目录和生成器工作区混用、运行口径漂移、环境不一致都会降低可执行性。
- 影响角色/系统：影响训练机操作者、版本产物、报告和后续发布。
- 建议测试点：训练目录初始化、proposal/embedder 冒烟训练、整链路评估结果归档。

### REQ-007

- 业务价值：自主训练不直接定义最终业务语义，但直接决定模型生产线的效率和持续迭代能力。
- 优先级理由：P1。统一求解包可以先由人工驱动的训练链路产出，但长期上必须有自动化闭环。
- 依赖关系：依赖生成器、训练、测试、评估、资产导出入口稳定，以及 `GoalContract/StudyContract`、结构化状态文件和受限 harness runtime。
- 风险点：如果只是把原始 CLI 直接暴露给模型，自主训练会退化成“自由 shell 代理”；如果没有 schema、watchdog 和晋级门，无人值守只会扩大失控成本。
- 影响角色/系统：影响项目维护者、长期训练成本、研究节奏和训练机安全边界。
- 建议测试点：goal 编译、study 恢复、schema 校验、fallback、停止规则、双任务隔离、无人值守演练。

### REQ-008

- 业务价值：数据治理和回灌机制把“偶尔能训练出一个好模型”变成“长期能维护一个稳定 bundle”。
- 优先级理由：P0。它决定模型版本是否真的可比较、可迭代。
- 依赖关系：依赖生成器、训练、测试、评估和版本策略。
- 风险点：
  - 若 `group1` 商业试卷继续强依赖类名，人审成本会长期维持在高位；
  - 若失败样本不按“切分/召回/混淆/分配/拒判”拆因，自动学习会无法精准调参；
  - 若测试集漂移、标签状态不清或回灌没有审计，会使版本比较失真。
- 影响角色/系统：影响项目维护者、自动训练、release 和验收。
- 建议测试点：冻结切分、版本追加、失败样本归档、错误归因和去重。

### REQ-009

- 业务价值：部署与操作文档是零基础执行能力和交付落地能力的保证，不是辅助材料。
- 优先级理由：P0。用户明确要求文档覆盖训练机、素材、生成器和自主训练部署。
- 依赖关系：依赖目录边界、CLI 稳定入口和 bundle 交付路径。
- 风险点：如果文档继续只讲训练，不讲最终 bundle 使用，就会继续强化错误定位；如果缺少成功标志和排障步骤，零基础用户仍然无法执行。
- 影响角色/系统：影响训练机操作者、交付使用者和项目交接。
- 建议测试点：训练机走查、bundle 安装走查、自主训练启动走查。

### REQ-010

- 业务价值：版本化发布把分散构建动作变成可审计、可回滚的正式交付流程。
- 优先级理由：P1。它不阻塞单次训练，但决定项目能否稳定交付和升级。
- 依赖关系：依赖训练产物、bundle、文档和评估结论齐备。
- 风险点：如果只有模型文件没有版本映射和交付清单，后续无法准确回滚；如果发布比较基于不同测试集，结论会失真。
- 影响角色/系统：影响项目维护者、发布流程、交付使用者和版本治理。
- 建议测试点：版本映射完整性、交付清单完整性、回滚演练。

### REQ-011

- 业务价值：一句业务目标能否稳定编译成合同，直接决定模型的主动性是“有目标地搜索”还是“无边界地乱试”。
- 优先级理由：P1。没有目标编译层，自主训练只能靠人工拼 study 配置，不构成真正无人值守。
- 依赖关系：依赖任务词典、指标字典、预算模板和合同 schema。
- 风险点：若目标编译结果含糊或不完整，后续每轮判断都会漂移；若允许执行阶段偷偷改合同，实验就不可比较。
- 影响角色/系统：影响项目维护者、控制器入口设计和后续所有 trial 的可比性。
- 建议测试点：goal compile、合同校验、结构化错误输出。

### REQ-012

- 业务价值：多 agent 交叉核查能放大模型的解释和比较能力，但前提是所有 I/O 都被 schema 和角色边界收口。
- 优先级理由：P1。它不替代控制器，但决定无人值守时能否在不扩大权限的前提下提升判断质量。
- 依赖关系：依赖 `Judge/Verifier/Reducer` 角色设计、decision schema 和 evidence pack。
- 风险点：如果多 agent 都能自由输出文本或命令，冲突和漂移会迅速放大；如果没有 reducer/fallback，study 会卡死在“意见不一致”。
- 影响角色/系统：影响 agent 集成层、decision protocol、fallback 和审计链路。
- 建议测试点：双 verdict 一致、双 verdict 冲突、schema 拒收、输入哈希一致性。

### REQ-013

- 业务价值：看门狗、自动验收和晋级门把“自动跑 trial”提升为“自动维护交付质量”，这是无人值守是否成立的最后一关。
- 优先级理由：P1。没有这层 gate，自主训练最多只是自动试验，不是可靠的模型工厂。
- 依赖关系：依赖评估口径、导出合同、solver 烟测和 watchdog 策略。
- 风险点：如果 `group1` 晋级仍主要看旧闭集训练分，不看整链路 business gate，候选模型会在真实业务面失败；如果看门狗不完善，长流程卡死会被误当成“仍在工作”。
- 影响角色/系统：影响控制器、release、solver 迁移主线和交付质量。
- 建议测试点：watchdog 触发、gate 阻断、solver smoke、promotion report。

### REQ-014

- 业务价值：只有把旧 `group1` 正式方案彻底清掉，实例匹配新方案才可能真正成为稳定的唯一主线。
- 优先级理由：P1。它不直接产生业务结果，但决定后续设计、开发、评估和发布会不会持续被双轨污染。
- 依赖关系：依赖新方案需求、设计、实现、测试和 cutover 计划齐备。
- 风险点：
  - 若保留旧 CLI 和旧用户文档，训练机操作者会继续走错入口；
  - 若保留旧测试与旧导出逻辑，回归结论会长期混入失真信号；
  - 若保留运行时代码兼容层，仓库复杂度会持续上涨。
- 影响角色/系统：影响项目维护者、实现者、训练机操作者、发布流程和文档可信度。
- 建议测试点：旧入口清单审计、grep 审计、发布资产审计、cutover 后全量回归。

### REQ-015

- 业务价值：调用方最直接的接入成本来自“输入预处理”；如果 solver 只接受本地路径，就会把业务系统常见的 `bytes/base64/URL` 输入强制转成临时文件，显著增加胶水代码和故障点。
- 优先级理由：P0。这是统一求解包可用性的基础能力，不是外围优化项。
- 依赖关系：依赖统一异常合同、输入适配层、图片解码库能力和网络安全策略（协议白名单、超时、大小上限）。
- 风险点：
  - “支持所有图片格式”如果没有工程边界，会变成不可验证口号；
  - 远程 URL 输入若无 SSRF/超限防护，会把 solver 变成隐性下载器风险面；
  - 同一图片经不同输入形态进入后结果不一致，会导致线上问题难定位。
- 影响角色/系统：直接影响调用方接入复杂度、solver 公开 API 稳定性、运行时安全边界和观测能力。
- 建议测试点：输入形态矩阵一致性、常见格式兼容性、下载超时与大小限制、结构化错误稳定性。

### REQ-016

- 业务价值：背景多样性不足会直接拉低生成样本的泛化质量；如果背景扩充仍依赖人工找图、人工筛坏图和人工拷贝到正式素材根，素材维护成本会持续偏高且容易把来源信息丢失。
- 优先级理由：P1。这不是 solver 主合同，但它直接影响训练数据多样性、素材治理效率和 generator 背景池质量。
- 依赖关系：依赖本地多模态模型、授权图片源、背景素材目录合同，以及可稳定执行的图片解码与落盘逻辑。
- 风险点：
  - 若把自动前景修补当成正式硬前置，质量不稳时会把风格判断引向错误背景；
  - 若每次任务中断后都要从头重做参考图分析和下载翻页，实际维护成本会迅速高于人工找图；
  - 若下载后没有解码/尺寸/重复质量门，正式背景池会持续混入坏图、小图和重复图；
  - 若合并逻辑直接重写整个素材根，容易破坏已有 `group1/group2` manifest 和现有 generator 素材包。
- 影响角色/系统：直接影响素材维护者、generator 背景池、训练数据多样性、授权来源审计和后续训练稳定性。
- 建议测试点：原图直送 VLM 风格分析、逐图分析 checkpoint 复用、逐图保底下载任务、下载任务恢复、损坏图拦截、小图拦截、重复图抑制、增量合并到正式背景根且不破坏其他 manifest。

### REQ-017

- 业务价值：本地多模态预标注往往运行时间长、请求成本高，如果每次中断都要整批重做，人工复核前的准备成本会迅速失控；按 `sample_id` 保存过程工件后，操作者才能稳定复用已完成样本并逐张排查异常。
- 优先级理由：P1。它不是 solver 最终业务合同，但它直接决定 `group1 prelabel-vlm` 是否真的可持续使用，而不是“实验性整批脚本”。
- 依赖关系：依赖 `group1 reviewed` 标签合同、`prelabel-vlm` 输出目录约定、本地 Ollama 请求协议和逐样本状态文件设计。
- 风险点：
  - 如果仍只保留聚合 `trace.jsonl`，恢复依据会弱到无法区分“整批成功一部分”与“最终文件写了一半”；
  - 如果把 `reviewed/*.json` 是否存在当成唯一跳过条件，会把“残缺草稿”误判成“已完成样本”；
  - 如果不按 `sample_id` 落原始请求/响应，人工排查 VLM 返回异常会退化成手工翻大 JSONL。
- 影响角色/系统：直接影响训练机操作者、人工复核人员、`group1` 预标注链路和后续回归测试设计。
- 建议测试点：同目录中断恢复、已完成样本跳过、过程目录完整性、逐样本错误重试、基于过程工件重建最终聚合文件。

## 总体判断

- 当前项目最重要的偏差不是“没有工具链”，而是“工具链抢占了主产品位置”。
- 本轮需求重构的核心纠偏是：
  - 把统一求解包/库提升为一级产品；
  - 把 generator/train/auto-train 降为支撑能力；
  - 把部署文档从训练机单线扩展到“训练机 + 生成器 + 自主训练 + bundle 使用”四条路径。
- 对 `group1` 而言，本轮新增的核心纠偏是：
  - 把“闭集类名识别”降级成素材治理辅助概念；
  - 把“实例匹配求解”提升为正式 solver 路线；
  - 把 query/scene 商业试卷和人工审核从“类名 + 框”切到“顺序 + 框”主线。
  - 把 `prelabel-vlm` 从“整批重跑脚本”提升为“逐样本可恢复、可审计的辅助预标注工具”。
- 对自主训练而言，新的纠偏重点不是“让模型更自由”，而是“把 harness 方法论正式化”：
  - 用目标合同、typed tools、typed observations 和 gate 把主动性转成可比较的搜索能力；
  - 用多 agent 交叉核查放大判断质量，而不是放大执行权限；
  - 用 watchdog 与 solver gate 把“自动跑”升级成“可无人值守”。
- 这一定义更符合当前代码演化方向：
  - 训练和评估主链路较完整；
  - 自主训练已有骨架；
  - 统一求解与 bundle 已有基础，但仍需提升为正式发布主线。
