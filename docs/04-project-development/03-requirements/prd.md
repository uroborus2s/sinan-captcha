## 产品背景

`sinan-captcha`（中文名：司南）当前仓库只有一份种子文档 [graphical_captcha_training_guide.md](../../../../graphical_captcha_training_guide.md)，尚无代码、数据集或训练环境说明。该文档已经明确：

- 当前问题不是 OCR 字符识别，而是图形验证码定位问题。
- 应拆成两个专项任务，而不是训练一个“万能模型”。
- 第一组更适合做多类别图标检测。
- 第二组更适合做滑块缺口定位，并可先用规则法做基线和自动标注辅助。

外部参考资料的结论如下：

- [dddd_trainer](https://github.com/sml2h3/dddd_trainer) 对 Windows/Linux、NVIDIA GPU 训练和新手安装流程友好，但核心任务是 OCR/CNN/CRNN 训练，不直接适用于当前图形定位任务。
- [captcha_trainer](https://github.com/kerlomz/captcha_trainer) 强调零基础、Windows GPU 和项目化管理，但其 README 明确当前模型场景仅支持图像分类，不适合作为本项目的直接模型路线。
- 因此，本项目借鉴上述仓库的“新手友好训练体验”，但模型任务定义采用目标检测路线，并以 Ultralytics YOLO 作为首选训练框架。

## 产品目标

- 让一个只有基础 IT 能力、没有模型训练经验的用户，能在一台 Windows + NVIDIA 显卡电脑上完成第一次训练闭环。
- 建立“样本导出 -> 真值校验 -> 自动标注/抽检 -> 数据质检 -> 模型训练 -> 验收复盘”的标准流程，尽量避免纯人工标框。
- 训练两个专项模型：
  - 第一专项模型：1 个多类别检测模型，用于第一组多图标顺序点击验证码。
  - 第二专项模型：1 个滑块缺口定位模型，用于第二组滑块验证码的缺口定位与偏移估计。
- 在需求阶段明确环境、数据、标注、训练、验收和迭代规则，为设计阶段和实现阶段提供稳定事实源。
- 为后续自主训练控制器固定职责边界、状态文件和 agent 接入方式，避免实现阶段边做边改。

## 用户角色

- 1. 零基础训练操作者
- 2. 项目维护者
- 3. 验证码生成服务维护者

## 本阶段交付

## 需求列表

### REQ-001 Windows 训练环境搭建

- 目标：让零基础用户在 Windows + NVIDIA 电脑上，于 1 个工作日内完成训练环境安装、自检与首次冒烟训练。
- 用户故事：作为零基础训练操作者，我希望按文档一步一步安装驱动、Python 和训练依赖，以便不需要理解底层原理也能开始训练。
- 前置条件：用户拥有 Windows 10/11、NVIDIA 显卡、管理员权限、至少 100GB 可用磁盘，以及一个可访问的自有/授权验证码样本来源。
- 主流程：1. 确认显卡和驱动可用；2. 安装 `uv` 和项目指定 Python 版本；3. 创建虚拟环境；4. 按 PyTorch 官方兼容矩阵安装 GPU 版 PyTorch；5. 安装 `ultralytics`、数据处理与标注工具依赖；6. 运行 `nvidia-smi`、`torch.cuda.is_available()` 和训练框架自检；7. 在文档中记录版本组合与常见报错处理。
- 异常/边界：若驱动、CUDA 和 PyTorch 版本不匹配，必须在环境自检阶段阻断后续流程；若显存不足 6GB，只允许先跑最小模型和小批量训练。
- 业务规则：环境管理统一使用 `uv`；首次落地不得要求用户手工编译复杂 C++ 扩展；版本组合必须可复现并写入正式文档。
- 数据约束：训练缓存、数据集和模型权重必须支持放在非系统盘；首次环境准备至少预留 80GB 存储空间。
- 权限/角色：零基础训练操作者按 runbook 执行；项目维护者批准标准版本组合；生成服务维护者不直接修改训练机环境。
- 验收标准：
  1. AC-1: 给定一台 Windows + NVIDIA 电脑，当用户按 runbook 执行安装，则能通过 GPU 自检并启动一次 10 分钟内可完成的冒烟训练。
  2. AC-2: 给定环境已经搭好，当用户重新进入项目目录时，则能在 10 分钟内恢复到可训练状态并开始继续训练。
- 优先级：P0
- 状态：已定义
- 关联设计：`docs/04-project-development/04-design/technical-selection.md` 中的训练环境与依赖管理章节。
- 关联测试：环境安装检查表、GPU 自检脚本、首次训练冒烟测试。

### REQ-002 样本获取与合规导出

- 目标：建立可重复执行的样本获取方案，并优先从自有或已授权的验证码生成链路直接导出标签，避免手工抄坐标。
- 用户故事：作为项目维护者，我希望从验证码生成端或已授权环境直接导出原图和坐标，以便快速积累训练数据且不依赖人工标注。
- 前置条件：存在可访问的验证码生成服务、渲染脚本、业务日志或其他合法数据来源。
- 主流程：1. 识别验证码类型为 `group1_multi_icon_match` 或 `group2_slider_gap_locate`；2. 导出图片、生成元数据和随机种子；3. 为第一组保存目标顺序、目标类别、目标框、干扰项框；4. 为第二组保存主图、滑块图、缺口目标框、中心点和偏移量；5. 对每条样本执行真值一致性校验与可重放校验；6. 只把通过校验的样本写入可追踪 JSONL 清单并归档原始图片。
- 异常/边界：若生成端拿不到最终真值，必须回退到“记录渲染参数再复算坐标并做重放校验”的方案；若仍无法证明真值正确，则该样本不得作为 `gold` 导出，也不得进入正式训练集；若样本来源不是自有/授权来源，则禁止纳入训练集。
- 业务规则：样本来源必须可审计；训练集、验证集、测试集必须按会话、模板或日期切分，防止同源图片泄漏到多个集合；生成器产出的训练样本只有在内部真值直出且通过一致性校验时，才允许标记为 `gold`；生成器应支持通过 preset 提升视觉难度，但第一版只允许像素级增强，例如阴影、背景模糊、边缘软化和 veil，不允许通过可配置参数改变标签几何语义。
- 数据约束：第一组首版目标数据量为训练集 5000 对、验证集 1000 对、测试集 1000 对；第二组首版目标数据量为训练集 3000 对、验证集 500 对、测试集 500 对；所有样本都要带唯一 ID 与来源标识。
- 权限/角色：生成服务维护者负责提供导出入口；项目维护者审核字段完整性；零基础训练操作者只消费已导出的样本。
- 验收标准：
  1. AC-1: 给定第一组和第二组验证码来源，当执行一次导出时，则能分别得到包含图片路径、类型、标签和来源元数据的样本清单。
  2. AC-2: 给定导出后的样本，当执行数据切分时，则训练/验证/测试集合之间不存在同源样本泄漏。
- 优先级：P0
- 状态：已定义
- 关联设计：数据流设计、数据目录设计、样本导出契约。
- 关联测试：样本导出字段校验、去重校验、训练/验证/测试切分校验。

### REQ-003 自动标注与低人工校正

- 目标：建立“自动生成标签为主、人工校正为辅”的标注流程，把人工工作量压缩到审核和纠错，而不是从零画框。
- 用户故事：作为零基础训练操作者，我希望系统先自动给出可用标签，我只需要检查和修正明显错误，以便在没有标注团队的情况下也能推进项目。
- 前置条件：已经有导出的原始样本，或至少能拿到未标注图片与查询图。
- 主流程：1. 优先把生成端内部真值直接作为 `gold` 标签；2. 对生成器产出的样本执行一致性校验、重放校验和负样本校验；3. 若样本不是生成器直出或缺少金标签，则对第一组使用规则分割查询图标 + 场景图模板/特征匹配生成预标注；4. 对第二组使用规则法生成滑块候选位置预标注；5. 将预标注导入 X-AnyLabeling 或 CVAT 批量复核；6. 把通过审核的数据标记为 `reviewed` 并回写正式数据集。
- 异常/边界：若某批样本的自动标注命中率低于 90%，则该批不得直接入库，必须进入人工复核或规则修正；若生成器直出的某条样本无法通过真值校验，则该样本必须直接阻断并重生，不允许降级为“先凑合训练”；若第一组出现大量未知图标类别，则必须发起变更评估是否需要引入匹配模型。
- 业务规则：每条标签都要有 `gold`、`auto` 或 `reviewed` 状态；自动标注只能作为预标注，不得在未抽检的情况下直接成为测试集真值；生成器输出的训练素材不得包含任何已知错误样本。
- 数据约束：原始图片不可覆盖；自动标注结果必须保留置信度、生成时间和使用的预标注方法；抽检集每批至少 200 张。
- 权限/角色：项目维护者定义标签状态流转规则；零基础训练操作者执行导入、复核与纠错；生成服务维护者补足生成端可直接提供的字段。
- 验收标准：
  1. AC-1: 给定一批未标注样本，当执行预标注流程时，则至少 90% 的样本能够自动生成候选框或点位。
  2. AC-2: 给定 200 张抽检样本，当人工复核自动标注结果时，则标签可用率达到 97% 以上后才能升级为 `reviewed`。
- 优先级：P0
- 状态：已定义
- 关联设计：自动标注流水线、标注工具选型、标签状态机。
- 关联测试：预标注精度抽检、标签状态流转校验、标注导入导出回归测试。

### REQ-004 数据集组织与质量治理

- 目标：把两个专项任务的数据结构、标签结构和版本规则固定下来，避免后续越训越乱。
- 用户故事：作为项目维护者，我希望数据目录、标签格式和版本命名都被统一规定，以便多人协作和后续追加样本时不破坏旧模型。
- 前置条件：已有样本导出与预标注能力。
- 主流程：1. 按任务拆分 `group1` 与 `group2`；2. 在每个任务下区分 `raw`、`interim`、`reviewed`、正式训练数据和 `reports`；3. 为第一组维护稳定的图标类别表并导出 YOLO 数据集；4. 为第二组维护滑块图案 mask、缺口定义、`master/tile/splits` paired dataset 和偏移字段；5. 生成 `dataset.yaml`（第一组）或 `dataset.json`（第二组）、版本号、样本统计和质检报告；6. 固定训练/验证/测试切分并归档。
- 异常/边界：第一组如果同一图标类别在查询图中可能重复出现，必须额外保存顺序或实例索引；若图标类别频繁变化超过维护阈值，则需要评估从多类别检测切换到检测 + 匹配方案；若第二组从单缺口滑块演化为多缺口或多目标，必须回到需求与设计阶段重评。
- 业务规则：数据集版本不可原地覆盖；新增样本必须追加新版本；测试集在一个迭代周期内保持冻结；任何未通过真值一致性校验的生成样本不得写入 `raw`；面向高级用户的 preset 覆盖文件只能放在生成器 workspace 的固定 `presets/` 目录，不允许依赖 `exe` 同级配置。
- 数据约束：第一组和第二组目录必须完全隔离；标签主事实源保留 JSONL，训练转换物可导出为 YOLO 文本格式；所有数据集版本都要产出统计摘要。
- 权限/角色：项目维护者维护类别表和版本策略；零基础训练操作者按固定脚本导入；生成服务维护者只提供原始样本和元数据。
- 验收标准：
  1. AC-1: 给定任何一个数据集版本，当新成员进入项目时，则能在 30 分钟内看懂目录结构、标签格式和切分规则。
  2. AC-2: 给定新增样本批次，当执行入库流程时，则不会覆盖旧版本且会自动生成统计摘要和版本号。
  3. AC-3: 给定同一批素材和随机种子，当切换 `firstpass` 与 `hard` 这类视觉难度 preset 时，则输出图片可以变化，但 `gold` 标签字段和几何语义不得因此漂移。
- 优先级：P0
- 状态：已定义
- 关联设计：数据目录规范、数据版本策略、标签转换设计。
- 关联测试：目录结构校验、标签 schema 校验、版本冻结校验。

### REQ-005 第一专项模型：多类别检测

- 目标：训练 1 个多类别检测模型，用于第一组“查询图给出多个图标，场景图中按顺序点击对应图标”的验证码。
- 用户故事：作为零基础训练操作者，我希望只训练一个清晰定义的多类别检测模型，以便先把第一组任务稳定跑通，而不是一开始就做复杂的多模型检索系统。
- 前置条件：第一组图标类别集合相对稳定，且样本中能拿到场景图、查询图、目标顺序和干扰图标信息。
- 主流程：1. 建立第一组类别表；2. 用场景图训练多类别目标检测模型；3. 用查询图读取目标顺序；4. 按类别把查询顺序映射到场景检测结果；5. 输出按顺序排列的点击点坐标；6. 记录失败样本和错序样本。
- 异常/边界：若同一类别在同一张图内出现多次，必须通过实例顺序、几何信息或额外规则完成消歧；若图标种类频繁变化且类别表维护成本过高，必须登记为设计阶段变更，而不是在需求阶段偷偷改任务定义。
- 业务规则：第一专项模型的首版路线固定为多类别检测；第一阶段不把“检测 + 相似度匹配”列为主交付，只保留为后备演进方向。
- 数据约束：第一组每条标注除了目标框，还必须包含干扰项框；查询顺序必须是正式字段，不得靠文件名猜测。
- 权限/角色：项目维护者维护类别映射和训练参数；零基础训练操作者负责按 runbook 执行训练、评估和错误复盘。
- 验收标准：
  1. AC-1: 给定第一组测试集，当模型推理完成时，则能输出按顺序排列的点击点坐标列表，而不只是检测框。
  2. AC-2: 给定冻结测试集，当评估第一专项模型时，则单目标点命中率不低于 92%，整组顺序全部命中率不低于 80%，平均点误差不高于 10 像素。
- 优先级：P0
- 状态：已定义
- 关联设计：第一专项模型架构、类别体系、查询顺序映射逻辑。
- 关联测试：第一组定位精度测试、顺序命中率测试、重复类别消歧测试。

### REQ-006 第二专项模型：滑块缺口定位

- 目标：训练 1 个滑块缺口定位模型，用于第二组“给定主图与滑块图，定位缺口目标并输出偏移量”的验证码。
- 用户故事：作为零基础训练操作者，我希望第二组任务先有一个简单、稳定的滑块定位目标，以便先跑通缺口定位和偏移估计，再讨论是否升级到分割或更复杂的轨迹建模方案。
- 前置条件：第二组样本能获取主图、滑块图、缺口目标框/中心点和横向偏移量，且目标语义在首版中保持单缺口。
- 主流程：1. 用规则法做第二组基线和预标注；2. 整理滑块定位数据集；3. 训练滑块缺口定位模型；4. 输出目标框、中心点和偏移量；5. 把规则法结果与模型结果做对比；6. 沉淀第二组失败样本。
- 异常/边界：若第二组目标不再是单缺口，或同一场景出现多个候选缺口，则必须重新评估任务定义；若规则法已经稳定满足业务门槛，可以把规则法保留为训练前 QA 和预标注工具，但不替代模型交付要求。
- 业务规则：第二专项模型首版以“滑块缺口定位模型”为正式交付；规则法只用于预标注、可行性确认和回归对照；生成器产出的第二组训练样本必须包含由内部真值直出的缺口位置。
- 数据约束：第二组标签最少包含 `master_image`、`tile_image`、`target_gap.bbox`、`target_gap.center`、`tile_bbox`、`offset_x` 和 `offset_y`；`master_image` 的缺口与 `tile_image` 必须由同一图案 mask 构造；测试集必须覆盖高亮目标、弱对比目标和复杂背景三类情况。
- 权限/角色：项目维护者定义第二组回归样本集合；零基础训练操作者执行规则法预标注、训练和评估。
- 验收标准：
  1. AC-1: 给定第二组测试集，当模型推理完成时，则能同时输出缺口目标框、中心点和横向偏移量。
  2. AC-2: 给定冻结测试集，当评估第二专项模型时，则点命中率不低于 95%，平均点误差不高于 8 像素，框 IoU 均值不低于 0.80，偏移误差均值不高于 6 像素。
- 优先级：P0
- 状态：已定义
- 关联设计：第二专项模型架构、规则法基线、输出契约。
- 关联测试：第二组点位精度测试、IoU 测试、偏移误差测试、规则法与模型对照测试。

### REQ-007 零基础操作手册

- 目标：把训练项目拆成小白可以执行的步骤，不要求用户先理解深度学习理论。
- 用户故事：作为零基础训练操作者，我希望有一份按步骤执行的操作手册，包括每一步看到什么、遇错怎么办，以便独立跑通项目。
- 前置条件：需求、数据和模型路线已经固定。
- 主流程：1. 先看“你需要准备什么”；2. 再看“环境怎么装”；3. 再看“样本从哪来”；4. 再看“如何自动标注和抽检”；5. 再看“训练命令怎么跑”；6. 再看“结果怎么看”；7. 再看“失败后怎么回滚和继续”。
- 异常/边界：文档不得默认用户理解 CUDA、显存、mAP、IoU 等概念；如果必须使用专业术语，必须在首次出现时给出通俗解释和判断标准。
- 业务规则：面向新手的文档要先讲准备条件、再讲操作、最后讲原理；每一步都要说明成功标志和失败时的处理入口。
- 数据约束：操作手册中的路径、目录和示例文件名必须与正式数据目录规范保持一致。
- 权限/角色：项目维护者维护手册；零基础训练操作者按手册执行并反馈卡点。
- 验收标准：
  1. AC-1: 给定一位只有基础 IT 能力的新用户，当其按手册执行时，则能在不阅读源码的情况下完成一次样本导出、预标注、训练和评估。
  2. AC-2: 给定用户遇到常见错误，当其按手册排查时，则能定位问题属于环境、数据、标注还是训练参数四类中的哪一类。
- 优先级：P0
- 状态：已定义
- 关联设计：用户指南、训练工作流页面、常见故障处理页面。
- 关联测试：文档走查、首次上手可用性验证、常见错误演练。

### REQ-008 验收、版本化与迭代回灌

- 目标：为两个专项模型建立统一的验收口径、产物清单和失败样本回灌机制，避免“训完就散”。
- 用户故事：作为项目维护者，我希望每次训练都留下可比较、可回滚、可继续优化的结果，以便模型迭代是可管理的工程活动，而不是一次性实验。
- 前置条件：已经有可复现的训练流程和冻结测试集。
- 主流程：1. 每次训练产出配置、日志、最佳权重和评估报告；2. 导出可部署前验证的模型文件；3. 汇总关键指标；4. 归档 Top 失败样本；5. 把失败样本加入下一轮候选数据池；6. 更新版本清单和结论摘要。
- 异常/边界：若训练结果优于旧版本但测试集或标签定义发生变化，则不得直接替换基线模型；若失败样本无法解释来源，则不得直接回灌。
- 业务规则：需求阶段只定义验收和产物要求，不进入线上部署实现；模型版本命名、测试集版本和训练配置必须可追溯。
- 数据约束：每个模型版本都要关联数据集版本、训练配置版本和评估报告；失败样本必须保留原图、预测结果和人工结论。
- 权限/角色：项目维护者负责批准基线版本；零基础训练操作者负责执行评估和归档；生成服务维护者负责支持补采新样本。
- 验收标准：
  1. AC-1: 给定一次完整训练，当训练结束时，则能产出模型文件、训练配置、评估报告和失败样本清单四类核心产物。
  2. AC-2: 给定两个历史版本，当比较新旧模型时，则能明确指出使用的数据版本、指标差异和是否满足替换条件。
- 优先级：P1
- 状态：已定义
- 关联设计：模型版本策略、评估报告模板、失败样本回灌流程。
- 关联测试：版本元数据校验、评估报告一致性校验、回灌样本去重校验。

### REQ-009 自主训练控制器与 opencode 接入

- 目标：在现有样本生成、训练、测试和评估 CLI 之上，建立一套可停、可续、可审计的自主训练控制器，并要求第一版先支持 `opencode` 作为 agent 运行时与 skill/command 容器。
- 用户故事：作为项目维护者，我希望系统能自动完成“生成样本 -> 训练 -> 测试 -> 评估 -> 判断下一步 -> 继续/放弃”闭环，同时把 AI 的职责限制在可解释、可审计的范围内，以便长时间训练时不依赖会话上下文记忆也能继续推进。
- 前置条件：`sinan-generator make-dataset`、`sinan train`、`sinan test`、`sinan evaluate` 已可独立执行；训练机具备本地或可访问的模型推理运行时；`opencode` 已安装并能在项目目录中加载本地 commands 与 skills。
- 主流程：1. 项目维护者定义 `study` 配置、预算和目标指标；2. Python 控制器初始化 `study` 目录和状态文件；3. 控制器按既定状态机调用生成、训练、测试和评估命令；4. 控制器把结果压缩为结构化摘要；5. 通过 `opencode` command/skill 触发 AI 读取摘要并输出结构化 JSON 决策；6. 控制器校验决策并执行 `RETUNE`、`REGENERATE_DATA`、`RESUME`、`PROMOTE_BRANCH` 或 `ABANDON_BRANCH`；7. 每轮 trial 都把输入、输出、结论和排行榜落盘；8. 达到停止条件后输出 study 总结。
- 异常/边界：若 AI 返回非合法 JSON 或与当前状态冲突，则控制器必须回退到规则决策，不得直接执行；若测试或评估报告缺失，则该 trial 不得晋级；若连续多轮无提升、预算耗尽或出现用户停止标记，则必须结束 study；若第一组与第二组的指标体系冲突，必须分开建 study，不允许混成一个总分数。
- 业务规则：第一版 `opencode` 只承担受限 agent 职责，不替代仓库内 Python 控制器；结果查看、结果压缩、评估判断、数据规划和 study 归档必须按 skill 或等价职责单元拆分；第一组与第二组共享控制器骨架，但拥有独立目标函数、独立停止规则和独立最佳模型账本；所有历史 trial 必须可通过磁盘状态文件重建，不依赖聊天上下文。
- 数据约束：每个 study 至少落盘 `study.json`、`best_trial.json`、`trial_history.jsonl`、`decisions.jsonl`；每个 trial 至少落盘 `input.json`、`dataset.json`、`train.json`、`test.json`、`evaluate.json`、`decision.json`、`summary.md`；所有决策都要记录使用的 agent/command、模型名、置信度和证据摘要。
- 权限/角色：项目维护者定义预算、阈值和目标；Python 控制器负责命令执行、状态写入和停止规则；`opencode` agent 负责读取摘要、调用 skill、输出结构化决策；零基础训练操作者只负责启动、暂停、恢复和查看结果，不直接改写 trial 状态文件。
- 验收标准：
  1. AC-1: 给定一份 `group1` 或 `group2` 的 study 配置，当启动自主训练时，则能创建独立 study 目录并完整跑通至少 1 个 trial 的“生成/训练/测试/判断”闭环。
  2. AC-2: 给定一个已运行多轮的 study，当会话上下文丢失或训练机重启后，则能仅根据状态文件恢复并继续执行，而不依赖历史聊天记录。
  3. AC-3: 给定一份结果摘要，当 `opencode` command/skill 产出判断时，则输出必须是合法 JSON，且只能落在 `RETUNE`、`REGENERATE_DATA`、`RESUME`、`PROMOTE_BRANCH`、`ABANDON_BRANCH` 五类动作内。
  4. AC-4: 给定两组专项同时存在，当自主训练运行时，则第一组和第二组必须拥有独立 study、独立排行榜和独立停止结论，不能相互覆盖。
- 优先级：P1
- 状态：已定义
- 关联设计：自主训练控制器设计、系统架构、模块边界、实施任务拆解。
- 关联测试：study 状态恢复测试、合法 JSON 决策测试、停止规则测试、group1/group2 双 study 隔离测试。

## 非功能需求

### NFR-001 可复现性

- 描述：从一台新 Windows + NVIDIA 电脑开始，按正式文档执行，应能在 1 个工作日内完成环境搭建并跑通首次训练冒烟流程。
- 度量口径：环境安装、自检和首次冒烟训练总耗时不超过 8 小时；关键版本信息和命令记录齐全。
- 验收方式：由非原作者按文档重新搭建一次环境并记录耗时和卡点。

### NFR-002 标注质量

- 描述：自动标注必须可控，不能因为追求省人工而引入大规模脏标签；生成器直出的 `gold` 样本不得包含任何已知错误。
- 度量口径：每批自动标注样本经 200 张抽检后，可用率不低于 97%；测试集标签必须全部经人工确认；生成器直出样本只有在通过一致性校验和重放校验后才允许进入数据集。
- 验收方式：抽检报告、标签状态统计、真值校验报告和问题样本复核记录。

### NFR-003 单机可运行性

- 描述：方案必须适配单台 Windows + NVIDIA 显卡电脑，不要求分布式训练。
- 度量口径：首版模型支持在 6GB 到 12GB 显存范围内通过调整模型规模和 batch size 跑通；500 张样本的冒烟训练应在 30 分钟内完成。
- 验收方式：在目标机器上执行冒烟训练并记录显存占用、耗时和成功率。

### NFR-004 合规性

- 描述：训练数据只能来自自有系统或已获授权的环境，不得使用来源不明的第三方验证码抓取数据。
- 度量口径：100% 样本都要记录来源和导出批次；来源不明样本不得进入正式训练集。
- 验收方式：样本来源清单、导出审计记录和数据入库审核记录。

### NFR-005 生成真值正确性

- 描述：受控生成器输出的训练素材必须 100% 来源于生成器内部真值，不允许依赖渲染后反推、人工猜测或未经验证的第三方接口返回值。
- 度量口径：100% 的 `gold` 样本都要记录 backend、随机种子、素材版本、配置版本和真值一致性校验结果；任一校验失败样本的入库率必须为 0；视觉难度增强参数只能影响像素表现，不能导致标签字段变化。
- 验收方式：批次清单、真值校验日志、重放报告和阻断记录。

### NFR-006 自主训练可控性与上下文隔离

- 描述：自主训练必须可审计、可恢复，并避免把长时训练链路绑定在单次聊天上下文上。
- 度量口径：100% 的 trial 决策都要有磁盘状态文件和结构化 JSON 记录；study 恢复时不需要读取历史聊天内容；AI 决策失败时必须在 1 次回退内切换到规则模式，不得导致 study 卡死。
- 验收方式：中断恢复演练、决策 JSON 校验记录、fallback 命中记录和 study 目录抽检。

## 范围外

- 当前阶段不实现线上推理服务、API 接口或网页前端。
- 当前阶段不训练“统一大杂烩模型”。
- 当前阶段不覆盖第三方未授权验证码数据抓取。
- 当前阶段不做移动端、Linux 集群或多机分布式训练。
- 当前阶段不采用“自由发挥式 AI shell 代理”直接接管训练机。

## 风险与约束

- 用户是否掌握验证码生成端源码，直接决定自动标注成本。
- 第一组图标类别是否稳定，决定多类别检测路线能否长期成立。
- 第二组是否始终保持单缺口滑块语义，决定滑块定位方案是否成立。
- Windows 驱动、CUDA 和 PyTorch 兼容性是新手最常见失败点。
- 若数据集背景模板过少，模型可能只记住背景而不是目标。
- 若 agent 权限边界和状态文件契约不先冻结，自主训练会退化为不可审计的长对话流程。
