## 产品背景

`sinan-captcha` 当前仓库已经具备 Go 生成器、Python 训练 CLI、自主训练控制器骨架和统一求解骨架，但此前公开文档的主叙事仍偏向“训练工程”。这会导致后续设计、发布和交付持续围绕内部产线，而不是围绕最终业务产物。

用户最新确认的最终业务目标是：

- 交付一个本地可调用的统一验证码求解包/库。
- `group2` 输入背景图和缺口图，输出背景图坐标系中的目标中心点。
- `group1` 输入查询图和背景图，按查询图中的图标顺序输出中心点序列。
- Go 生成器、Windows 训练 CLI 和自主训练 CLI 都必须围绕这个最终交付物服务。

## 产品定位

- 一级产品：统一验证码求解包/库
- 二级产品：模型生产工具链
  - `sinan-generator`
  - `sinan train/test/evaluate/release`
  - `sinan auto-train`
- 目标交付方式：
  - 1 个可复制的模型 bundle 目录
  - 1 套统一业务语义
  - 1 个对外调用入口

## 需求列表

### REQ-001 统一求解包与单一调用入口

- 目标：把 `group1` 与 `group2` 的求解能力收口为 1 个统一求解包/库和 1 套稳定调用合同。
- 用户故事：作为调用方，我希望只加载 1 个 bundle 并调用 1 个入口，就能得到 `group1` 或 `group2` 的求解结果，以便不需要理解内部子模型编排。
- 前置条件：对应任务的模型、配置和版本信息已经冻结并可打包。
- 主流程：
  1. 加载统一求解包和指定 bundle。
  2. 接收 1 份任务请求。
  3. 按显式 `task_hint` 或确定性输入形态规则路由到 `group1` 或 `group2`。
  4. 执行对应求解链路并返回统一结构化响应。
- 异常/边界：若请求同时携带两类任务输入，必须直接报错；若 `task_hint` 与输入形态冲突，必须直接报错；不得要求调用方自行协调多个子模型。
- 业务规则：对外只保留 1 个正式入口；统一入口只暴露业务语义，不泄露内部训练目录和子模型结构。
- 数据约束：响应必须包含 `request_id`、`task`、`status`、`bundle_version` 和任务专属结果或结构化错误对象。
- 权限/角色：项目维护者冻结统一合同；调用方只消费统一入口；训练机操作者不直接修改对外合同。
- 验收标准：
  1. AC-1: 给定同一个求解包和 bundle，当调用方分别提交 `group1` 与 `group2` 请求时，则都能通过同一入口得到结构化结果。
  2. AC-2: 给定一份输入冲突的请求，当统一入口执行时，则必须返回结构化错误而不是尝试混合求解。
- 优先级：P0
- 状态：已定义
- 关联设计：统一求解服务、请求/响应合同、路由规则设计。
- 关联测试：统一入口测试、路由冲突测试、结构化错误合同测试。

### REQ-002 `group2` 拖动式验证码求解能力

- 目标：提供面向拖动式验证码的正式求解能力，输出背景图坐标系中的目标中心点。
- 用户故事：作为调用方，我希望输入背景图和缺口图后，直接拿到目标中心点，以便驱动后续拖动或验证逻辑。
- 前置条件：`group2` 模型已经完成训练、评估并通过验收。
- 主流程：
  1. 接收背景图和缺口图。
  2. 运行 `group2` 定位模型。
  3. 输出目标中心点，并按需要附带目标框、偏移量和置信度。
- 异常/边界：若缺少必需图片、图片损坏或尺寸不合法，必须返回结构化错误；若只具备中心点，不得伪造完整拖动轨迹。
- 业务规则：`group2` 的主业务结果固定为背景图坐标系中的中心点；偏移量和目标框属于辅助结果，不得替代中心点字段。
- 数据约束：输入图片必须与请求 ID 绑定；输出点位必须带坐标系说明；若提供偏移量，必须说明其参考坐标。
- 权限/角色：调用方消费结果；项目维护者定义合同和验收阈值；训练机操作者负责产出满足要求的模型。
- 验收标准：
  1. AC-1: 给定一张背景图和一张缺口图，当求解成功时，则响应中必须包含目标中心点和状态字段。
  2. AC-2: 给定需要辅助调试或回放的场景，当启用详细结果模式时，则能额外返回目标框、偏移量或置信度等可解释字段。
- 优先级：P0
- 状态：已定义
- 关联设计：`group2` 推理合同、定位结果封装、bundle 目录设计。
- 关联测试：`group2` 响应合同测试、点位正确性测试、错误输入测试。

### REQ-003 `group1` 点选式验证码求解能力

- 目标：提供面向点选式验证码的正式求解能力，输出按查询顺序排列的中心点序列。
- 用户故事：作为调用方，我希望输入查询图和背景图后，能直接拿到按顺序排列的点击点，以便不需要自行匹配查询图标和场景图标。
- 前置条件：`group1` 的 `query splitter`、`scene proposal detector`、`icon embedder` 和 `matcher` 已完成训练、配置冻结和联调。
- 主流程：
  1. 接收查询图和背景图。
  2. 解析查询图中的目标图标顺序并切出 query 项。
  3. 在背景图中检测全部图标候选位置，但不以闭集类名作为主判断。
  4. 对 query 项和 scene 候选做向量编码、相似度匹配和全局分配。
  5. 输出有序中心点序列。
- 异常/边界：若查询图顺序无法解析、候选检测不足、相似度分配存在明显歧义或最终结果缺少必要点位，必须返回结构化状态而不是静默输出伪结果。
- 业务规则：
  - `group1` 的业务结果是有序中心点序列；
  - 不得只输出无序候选框；
  - 不得把闭集类名表作为 solver 成功求解的前置条件；
  - 顺序状态、歧义状态和拒判状态必须可解释。
- 数据约束：
  - 输出序列长度必须与查询图中目标项数量一致，除非响应明确声明失败或歧义；
  - 调试信息允许包含候选框、相似度、分配结果和歧义原因；
  - `class_name` 只能作为辅助元数据，不能作为正式业务合同主字段。
- 权限/角色：调用方只消费顺序结果；项目维护者定义匹配规则和异常状态；训练机操作者负责产出可用模型。
- 验收标准：
  1. AC-1: 给定一张查询图和一张背景图，当求解成功时，则响应中必须包含按查询顺序排列的中心点序列。
  2. AC-2: 给定存在重复图标、相似图标或未知语义图标的场景，当求解无法稳定消歧时，则响应中必须显式标记歧义或失败状态。
  3. AC-3: 给定素材库中新加入但不在旧类名表中的图标，当它已进入新素材和训练体系后，则 solver 仍可通过实例匹配完成求解，而不是因为缺少类名直接失效。
- 优先级：P0
- 状态：已定义
- 关联设计：`group1` 双模型流水线、匹配规则、统一响应合同。
- 关联测试：顺序命中测试、重复类别歧义测试、结构化状态测试。

### REQ-004 统一 bundle、manifest 与交付包

- 目标：把求解包运行所需的模型、配置、类别表和版本信息打包成 1 个可迁移的 bundle，并形成可交付包。
- 用户故事：作为交付使用者，我希望把 1 个 bundle 目录复制到新机器后就能完成加载和校验，以便不依赖原训练目录结构。
- 前置条件：`group1`、`group2` 模型和相关配置已冻结。
- 主流程：
  1. 生成 bundle manifest。
  2. 按统一目录收口模型、配置、标签映射和版本元数据。
  3. 在打包前执行完整校验。
  4. 生成交付包并附带必要说明。
- 异常/边界：若 manifest 缺字段、路径非法或模型文件缺失，必须在校验阶段阻断；不得依赖训练目录的绝对路径。
- 业务规则：bundle 目录必须自描述；交付包必须能和训练目录解耦；对外交付不得要求使用者手工拼装模型文件。
- 数据约束：bundle 至少包含 manifest、模型文件、配置文件、版本映射和必要的标签/类别元数据。
- 权限/角色：项目维护者负责冻结版本和目录规范；交付使用者只消费 bundle；训练机操作者不得手工改写 bundle 内结构。
- 验收标准：
  1. AC-1: 给定一个完整 bundle，当把它复制到另一台机器或新路径后，则仍能通过 manifest 相对路径成功加载。
  2. AC-2: 给定一个缺失模型文件或非法路径的 bundle，当执行校验时，则必须阻断并返回明确错误。
  3. AC-3: 给定一次正式发布，当交付包生成完成时，则必须包含求解包、bundle、版本信息和调用说明。
- 优先级：P0
- 状态：已定义
- 关联设计：bundle 目录设计、release 设计、部署设计。
- 关联测试：跨路径加载测试、manifest 校验测试、交付包完整性测试。

### REQ-005 Go 生成器 CLI 与 gold 样本导出

- 目标：提供一个受控生成器，为 `group1` 和 `group2` 生成可训练样本，并输出可审计的真值与 QA 结果。
- 用户故事：作为项目维护者，我希望通过固定工作区、素材包和 preset 生成高质量样本，以便稳定为最终 bundle 生产训练数据。
- 前置条件：已具备背景素材、图标素材和合法的数据来源。
- 主流程：
  1. 初始化生成器工作区。
  2. 导入或抓取素材并建立索引。
  3. 按任务与 preset 生成样本。
  4. 执行真值校验、重放校验和批次 QA。
  5. 导出 `group1` 和 `group2` 正式训练数据集目录。
- 异常/边界：若素材损坏、真值无法证明正确或 QA 失败，必须阻断样本出库；不得把未经验证的样本标为 `gold`。
- 业务规则：
  - 生成器拥有训练样本几何真值的定义权；
  - 视觉难度增强只能改变像素表现，不得改变标签几何语义；
  - `group1` 素材库必须以 `asset_id` 为主键管理，语义分类只作为素材治理辅助，不再作为 solver 主合同。
- 数据约束：
  - 所有样本都要带唯一 ID、来源信息、随机种子、标签状态和 QA 结论；
  - `group1` 真值必须同时保留 `asset_id`、`template_id`、`variant_id` 等实例身份信息；
  - `group1` 导出契约必须至少覆盖：
    - 类无关候选检测数据
    - 图标匹配/嵌入学习数据
    - 端到端评估数据
  - `group2` 仍维持 paired dataset 契约。
- 权限/角色：生成服务维护者提供素材和合法数据来源；项目维护者维护 preset 与 QA 规则；训练机操作者只消费已导出的数据集。
- 验收标准：
  1. AC-1: 给定一套合法素材和 preset，当执行 `make-dataset` 时，则能导出 `group1` 或 `group2` 的正式数据集目录、清单和 QA 结果。
  2. AC-2: 给定存在坏素材、坏标签或 QA 失败的批次，当导出流程执行时，则必须阻断并给出明确失败原因。
  3. AC-3: 给定 `group1` 的一个新图标素材，当它入库后，则必须先得到稳定的 `asset_id` 与来源记录，再进入生成和训练链路。
- 优先级：P0
- 状态：已定义
- 关联设计：生成器设计、数据导出契约、QA 设计。
- 关联测试：真值校验测试、坏素材阻断测试、数据集导出测试。

### REQ-006 Windows 训练 CLI、测试与评估闭环

- 目标：在 Windows + NVIDIA 单机上提供正式训练闭环，把数据集训练成可用于 bundle 的模型产物。
- 用户故事：作为训练机操作者，我希望按统一命令在训练目录里完成环境初始化、训练、测试和评估，以便稳定产出可交付模型。
- 前置条件：训练机具备 Windows、NVIDIA GPU、`uv`、Python 和项目要求的训练依赖。
- 主流程：
  1. 初始化独立训练目录。
  2. 导入 `group1` 或 `group2` 数据集版本。
  3. 执行训练、预测、测试和评估。
  4. 生成训练报告、评估结果和可发布模型。
- 异常/边界：若环境不满足要求、数据契约非法或评估失败，不得进入正式发布流程；训练目录不得和生成器工作区混用。
- 业务规则：训练目录、生成器工作区和源码仓库必须分离；对外正式运行口径统一使用 `uv run sinan ...`。
- 数据约束：
  - 每次训练都必须关联数据集版本、训练名、配置、报告和模型权重；
  - `group1` 训练链必须支持分别训练和验证：
    - `scene proposal detector`
    - `icon embedder`
    - `matcher` 校准参数
  - `group1` 的正式评估必须以整链路业务结果为主，不得只以闭集分类指标充当最终结论。
- 权限/角色：训练机操作者执行训练；项目维护者冻结版本和验收规则；调用方不直接使用训练目录。
- 验收标准：
  1. AC-1: 给定一台符合要求的 Windows + NVIDIA 训练机，当按 runbook 执行时，则能初始化训练目录并完成一次冒烟训练。
  2. AC-2: 给定一次正式训练，当训练结束后，则能产出模型文件、测试报告、评估结果和版本化目录。
- 优先级：P0
- 状态：已定义
- 关联设计：训练 CLI 设计、训练目录规范、评估与测试设计。
- 关联测试：训练目录初始化测试、训练冒烟测试、评估合同测试。

### REQ-007 自主训练 harness、受限 agent 与无人值守闭环

- 目标：在现有生成、训练、测试和评估能力之上，建立 harness-first 的自主训练闭环，把大模型的主观能动性限制在封闭动作空间内，同时让训练过程具备真正的无人值守、可恢复和可审计能力。
- 用户故事：作为项目维护者，我希望系统能把一句业务目标自动编译成可执行 study，并持续完成“规划 -> 建数 -> 训练 -> 测试 -> 判断 -> 自动验收 -> 下一轮”的循环，以便尽量减少人工盯盘，同时又不让模型越权接管训练机。
- 前置条件：生成器、训练、测试、评估和资产导出入口已稳定；本地控制器、`opencode` commands/skills、JSON Schema 校验和规则 fallback 可用。
- 主流程：
  1. 维护者提交一句业务目标或结构化目标。
  2. 系统将目标编译为 `GoalContract` 与 `StudyContract`，冻结预算、目标指标、允许动作集和停止规则。
  3. Python 控制器按显式状态机驱动 stage capsules，并通过受限 tool adapter 调用 `sinan-generator` 与 `sinan` CLI。
  4. 每个阶段的长输出被压缩成 `ObservationPacket` 与 `TrialEvidencePack`，供 agent 判断使用。
  5. `Judge` 与 `Verifier` 在封闭动作集内输出结构化 verdict。
  6. 控制器只在 verdict、schema、输入哈希和策略门全部通过时执行 `KEEP_TRAINING`、`RETUNE`、`REGENERATE_DATA`、`RETEST`、`STOP` 或 `PROMOTE_CANDIDATE`。
  7. 系统持续循环，直到达到目标指标、命中停止条件，或通过自动验收晋级为候选交付资产。
- 异常/边界：若 agent 输出非法 JSON、schema 不匹配、输入哈希不一致、角色 verdict 冲突或 watchdog 超时，控制器必须拒收该输出并回退到规则模式；上下文丢失不得导致 study 无法恢复；不得让 agent 直接调用原始训练 shell。
- 业务规则：

### REQ-012 严格 Schema I/O、角色分工与多 agent 交叉核查

- 目标：让所有 agent 输入输出都遵循严格 schema，并通过角色分工和交叉核查放大主观能动性而不放大失控风险。
- 用户故事：作为项目维护者，我希望模型可以主动解释结果、提出策略、互相核查，但每一步都必须遵循严格 I/O 规范，以便无人值守运行时不被模糊文本拖垮。
- 前置条件：`Judge`、`Verifier`、`Reducer` 角色和各自 schema 已冻结。
- 主流程：
  1. 控制器向 `Judge` 提供 `TrialEvidencePack`。
  2. `Judge` 输出结构化 `JudgeVerdict`。
  3. 控制器向 `Verifier` 提供同一证据和 `JudgeVerdict`。
  4. `Verifier` 输出结构化 `VerifierVerdict`。
  5. 控制器或 `Reducer` 只在 verdict 合法且不冲突时产出最终 `DecisionEnvelope`。
- 异常/边界：任一角色输出非法 JSON、越权动作、额外字段、输入哈希不一致或 verdict 冲突时，必须拒收并回退到规则模式；不得依赖长自由文本继续推理。
- 业务规则：多 agent 的价值在于交叉核查和策略比较，不在于让多个 agent 并行自由执行工具；控制器只接受封闭动作集内的 verdict。
- 数据约束：所有被接受的 verdict 都要记录 `schema_version`、`role`、`allowed_action`、`reason_codes`、`input_hash`、`confidence` 和 `evidence_refs`。
- 权限/角色：`Judge` 负责动作建议；`Verifier` 负责合规和证据一致性审查；`Reducer` 负责归并或拒收；控制器拥有最终执行权。
- 验收标准：
  1. AC-1: 给定同一份 trial 证据，当 `Judge` 与 `Verifier` 都输出合法 verdict 时，则控制器能生成最终 `DecisionEnvelope`。
  2. AC-2: 给定一次非法 verdict 或冲突 verdict，当控制器处理时，则必须拒收并进入 fallback，而不是继续执行该动作。
- 优先级：P1
- 状态：已定义
- 关联设计：harness 角色设计、decision schema、fallback 设计。
- 关联测试：schema 校验测试、双 verdict 冲突测试、reducer 测试。

### REQ-013 无人值守看门狗、自动验收与晋级门

- 目标：为真正的无人值守训练建立看门狗、自动验收和晋级门，确保系统只会推广在真实业务与 solver 验证上都达标的候选模型。

  1. 调用方提交图片输入（本地路径、`bytes`、base64/data URI 或 `http(s)` URL）。
  2. solver 输入适配层按输入类型执行解析、下载（如需要）和解码。
  3. 解码结果统一归一到内部图像对象后进入 `group1/group2` 求解链路。
  4. 输出保持既有业务结果合同，不向调用方暴露内部临时文件或下载细节。
- 异常/边界：若输入类型不受支持、图片无法解码、网络下载超时/超限/协议非法，必须返回结构化 `SolverInputError`；不得把异常吞掉后继续推理。
- 业务规则：
  - 对外仍保持 `sn_match_slider(...)` 与 `sn_match_targets(...)` 的语义稳定；
  - 输入兼容增强不得改变 `group1/group2` 的业务结果字段；
  - 网络输入必须受安全策略约束（协议白名单、大小上限、超时、重定向限制）。
- 数据约束：
  - 输入适配层必须记录输入类型、解码结果和失败原因（可审计）；
  - 若输入为网络 URL，必须可配置是否启用远程拉取，以及最大文件大小与请求超时；
  - “支持所有图片格式”的工程口径为：支持运行时解码器（Pillow）可稳定解码的格式族，并对不支持格式返回确定性错误。
- 权限/角色：项目维护者冻结输入合同和安全阈值；实现者负责适配层与回归测试；调用方只需按公开输入口径传参。
- 验收标准：
  1. AC-1: 给定本地路径、`bytes`、base64/data URI 和 `http(s)` URL 四类输入，当调用同一函数时，则都能得到与本地路径等价的业务结果结构。
  2. AC-2: 给定损坏图片、非法 URL、超限文件或超时下载场景，当执行求解时，则必须返回结构化输入错误并附带可定位原因。
  3. AC-3: 给定 JPEG、PNG、WebP、BMP、GIF、TIFF 等常见格式，当格式可被运行时解码器识别时，则 solver 能成功进入求解流程。

- 状态：已定义

### REQ-016 背景素材风格扩充与正式 backgrounds 合并

- 目标：让训练仓库 CLI 能基于给定参考图直接分析背景风格、从授权图片源下载相似背景图，并把通过质量门的结果并入正式 `backgrounds/` 素材根。

  6. 下载结果经过图片解码、尺寸门槛和去重校验后落入 `incoming/backgrounds/`。
  7. 若任务中断，下一次在同一 `output-root` 重跑时必须复用已完成的逐图分析结果和下载任务状态，而不是从头重做。

  - 若任务中断，已完成的参考图分析记录和已完成的下载任务状态不得丢失；恢复时只能重试未完成部分。
  - 网络下载只能使用已授权来源，不得扩展到来源不明的抓取站点。
- 业务规则：
  - V1 以“原图直送多模态模型 + 提示词忽略前景干扰”为正式策略，不把自动修补背景作为必选前置步骤。
  - 逐图分析、风格汇总和下载任务状态都必须持久化到 `output-root/reports/`，支持同目录自动恢复。
  - 每张参考图至少要生成 1 个保底下载任务；汇总搜索词只能作为补充扩充任务，不能替代逐图保底任务。
  - 正式落盘的背景图必须通过最小质量门：可解码、满足尺寸阈值、未命中重复策略。
  - 合并到正式素材根时不得破坏已有 `group1/group2` manifest；本轮只增量写入 `backgrounds/` 和背景索引。
- 数据约束：
  - 每张新背景图必须记录 `provider/query/author/license/source_url/file_name` 等来源字段。
  - 报告必须记录风格画像、原始模型输出、逐图分析 checkpoint、汇总结果、下载任务状态、下载成功/失败/跳过原因、质量门结果和正式合并结果。
  - 去重策略必须至少覆盖同批下载集和目标素材根中的已有背景图。
- 权限/角色：项目维护者冻结背景扩充边界与质量阈值；实现者负责 CLI、质量门、去重与合并逻辑；素材维护者只通过正式命令扩充和合并背景图。
- 验收标准：
  1. AC-1: 给定包含验证码图标或缺口干扰的参考背景图，当执行背景扩充命令时，则系统仍能直接基于原图生成背景风格摘要和英文搜索词，而不要求先手工修图。
  2. AC-2: 给定任务在逐图分析或下载阶段中断时，当操作者在同一 `output-root` 重跑命令，则系统会复用已完成的参考图分析结果和已落盘的下载状态，只继续未完成部分。
  3. AC-3: 给定多张参考背景图且未设置更小的全局 `limit` 时，当执行背景扩充命令，则系统至少为每张参考图下载 1 张通过质量门的相似背景图。

### REQ-017 `group1 prelabel-vlm` 逐样本过程工件与断点续传

- 目标：让 `uv run sinan train group1 prelabel-vlm` 在本地多模态预标注场景下具备逐样本可恢复、可跳过已完成样本、可按文件名审计原始返回的正式运行语义。
- 用户故事：作为训练机操作者，我希望在同一 `--project` 目录下重跑 `prelabel-vlm` 时，只继续未完成样本，并能按 `sample_id` 查看请求、原始返回和归一化结果，以便降低本地大模型预标注的时间成本和排障成本。
- 前置条件：`query/` 与 `scene|scence/` 成对图片目录可用，本地 Ollama 多模态模型可达，`group1 reviewed` 合同已冻结。
- 主流程：
  1. 系统扫描同名 `query + scene|scence` 图片对，并在 `--project` 下创建正式过程目录 `process/`。
  2. 系统为每个 `sample_id` 建立 `process/samples/<sample_id>/` 目录，并在调用模型前先写入 `status.json` 与 `request.json`。
  3. 系统在收到模型结果后，按样本分别写入 `response.json`、`normalized.json` 和必要的 `error.json`。
  4. 当同一 `--project` 目录再次执行时，系统必须先读取 `process/index.json` 与逐样本状态，只重跑未完成样本。
  5. 已完成样本必须直接复用已有过程工件，不得重复调用大模型。
  6. 全部样本完成后，系统基于逐样本过程工件重建 `reviewed/*.json`、`labels.jsonl`、`trace.jsonl` 和 `summary.json`。
- 异常/边界：
  - 仅存在 `reviewed/*.json` 不得单独作为样本“已完成”的正式依据，必须同时具备合法状态与归一化结果工件。
  - 若 `status.json`、`normalized.json` 或关键过程文件缺失、损坏或互相不一致，则该样本只能视为 `partial` 或 `failed`，不得跳过。
  - 若单样本失败，系统必须保留该样本错误信息，并允许后续只重试该样本，而不是强制整批重做。
  - 最终仍必须经过人工复核，再执行 `exam export-reviewed`；本需求不把 VLM 输出升级为正式真值。
- 业务规则：
  - `process/` 是 `prelabel-vlm` 的正式恢复入口和审计入口，不能仅依赖聚合 `trace.jsonl` 做恢复。
  - 正式过程目录必须按 `sample_id` 分组，支持按文件名直接检查请求、原始响应和归一化结果。
  - 已完成样本在同一 `--project` 目录重跑时，默认必须跳过，不得再次消耗大模型推理成本；只有显式强制重跑时才允许覆盖。
  - 最终人工复核输出格式仍保持 `query=query_item`、`scene=NN`，可选类别提示继续保留在附属元数据中。
- 数据约束：
  - `--project/process/index.json` 必须记录运行级元数据、样本总数、状态统计和恢复所需索引。
  - `--project/process/samples/<sample_id>/` 至少保留 `status.json`、`request.json`、`response.json`、`normalized.json`、`error.json` 这五类过程文件中的适用子集。
  - 只有 `status=completed` 且 `normalized.json` 完整存在时，样本才允许在后续重跑中被判定为已完成。
  - `labels.jsonl`、`trace.jsonl` 与 `summary.json` 必须可由逐样本过程工件确定性重建。
- 权限/角色：项目维护者冻结恢复语义和过程目录合同；实现者负责状态机、逐样本工件和回归测试；训练机操作者只通过正式 CLI 启动、恢复和检查任务。
- 验收标准：
  1. AC-1: 给定一批图片对在执行到一半时中断，当操作者在同一 `--project` 目录重跑命令，则系统只继续未完成样本，已完成样本不会再次请求大模型。
  2. AC-2: 给定某个 `sample_id` 已存在 `status=completed` 且 `normalized.json` 完整，当再次运行同一任务时，则该样本会被直接复用，并保留原有请求/响应过程文件。
  3. AC-3: 给定任一 `sample_id` 的模型调用成功或失败，当任务结束后，则都能在 `process/samples/<sample_id>/` 下找到对应的状态文件和原始过程工件，用于按文件名检查。
  4. AC-4: 给定同一 `--project` 目录内已有逐样本过程工件，当全部样本完成时，则系统能基于这些工件重建 `reviewed/*.json`、`labels.jsonl`、`trace.jsonl` 与 `summary.json`。
- 优先级：P1
- 状态：已定义
- 关联设计：`group1` 实例匹配重构设计、`prelabel-vlm` 过程目录设计、人工复核前预标注审计设计。
- 关联测试：逐样本恢复测试、已完成样本跳过测试、过程工件落盘测试、聚合结果重建测试。

## 非功能需求

### NFR-001 业务正确性

- 描述：`group1` 与 `group2` 的正式业务结果都必须以中心点坐标表达，并满足对应验收阈值。
- 度量口径：
  - `group1` 至少达到 `full_sequence_hit_rate >= 0.93`、`single_target_hit_rate >= 0.985`、`mean_center_error_px <= 6.0`
  - `group2` 至少达到 `point_hit_rate >= 0.93`、`mean_iou >= 0.85`、`mean_center_error_px <= 8.0`
- 验收方式：固定测试集评估报告、任务级指标报告和失败样本复核记录。

### NFR-002 可复现性

- 描述：从一台新 Windows + NVIDIA 电脑开始，按正式文档执行，应能在 1 个工作日内完成环境搭建并跑通首次冒烟训练。
- 度量口径：环境安装、自检和首次冒烟总耗时不超过 8 小时；关键版本、命令和目录记录齐全。
- 验收方式：由非原作者按文档重走一次训练机部署流程。

### NFR-003 单机可运行性

- 描述：首版方案必须适配单台 Windows + NVIDIA 显卡电脑，不要求分布式训练。
- 度量口径：支持在 6GB 到 12GB 显存范围内通过调整模型规模和 batch size 跑通；500 张样本的冒烟训练在 30 分钟内完成。
- 验收方式：在目标机器上记录显存占用、耗时和成功率。

### NFR-004 数据合规性

- 描述：训练样本只能来自自有系统或已获授权环境，不得使用来源不明的第三方抓取数据。
- 度量口径：100% 样本记录来源、导出批次和使用状态；来源不明样本入库率为 0。
- 验收方式：来源审计记录、数据入库审核记录。

### NFR-005 gold 真值正确性

- 描述：生成器直出的 `gold` 样本必须来源于内部真值并通过一致性校验，任一校验失败样本不得进入正式数据集。
- 度量口径：100% `gold` 样本记录 backend、随机种子、素材版本、配置版本和真值校验结果；失败样本入库率为 0。
- 验收方式：真值校验日志、重放报告、阻断记录。

### NFR-006 自主训练可控性与可恢复性

- 描述：自主训练必须可停、可续、可审计，且不依赖长会话上下文。
- 度量口径：100% trial 决策落盘；训练机重启后能恢复；agent 决策失败时必须在 1 次回退内进入规则模式。
- 验收方式：中断恢复演练、非法 JSON fallback 演练、study 目录抽检。

### NFR-007 bundle 可迁移性

- 描述：bundle 必须整体可复制、相对路径加载，并支持启动前完整校验。
- 度量口径：bundle 迁移到新绝对路径后仍能 100% 加载；完整校验在 5 秒内完成；缺失文件必须首次校验即阻断。
- 验收方式：跨路径加载测试、缺失文件测试、manifest 校验测试。

### NFR-008 文档可执行性

- 描述：文档必须让零基础操作者和交付使用者在无口头说明的情况下完成部署与冒烟。
- 度量口径：训练机部署、新 bundle 安装和自主训练启动都必须有逐步操作、成功标志和常见失败处理；文档走查通过率 100%。
- 验收方式：训练机部署演练、bundle 安装演练、文档走查记录。

### NFR-009 Harness 自治有效性与无人值守可靠性

- 描述：自主训练 harness 必须在固定预算下放大模型的策略主动性，但不能放大越权风险；系统只接受通过 schema、证据和 gate 的结果。
- 度量口径：100% 被接受的 agent 输出都通过 schema 校验并带输入哈希；单次 study 可在无人工修改状态文件的前提下连续运行至少 12 小时；100% 被晋级候选都通过 solver gate 且主指标不低于当前最佳基线。
- 验收方式：长流程无人值守演练、schema 审计、promotion gate 演练。

### NFR-010 单一正式方案与代码清洁性

- 描述：`group1` 重构完成后，仓库中只能保留 1 套正式主线实现，不得长期保留闭集类名旧方案的兼容层、并行 CLI 或并行文档。
- 度量口径：
  - 100% 正式 `group1` 用户文档、设计文档和发布资产只描述新方案；
  - 100% 正式 CLI 入口只指向新方案；
  - 旧方案运行时代码、旧方案测试和旧方案正式文档残留率为 0。
- 验收方式：代码审计、CLI 清单审计、文档 grep 审计、发布物清单审计。

## 范围外

- 当前范围不包含公网 HTTP 平台、多租户鉴权和在线扩缩容。
- 当前范围不采用“自由发挥式 AI shell 代理”替代受控控制器。
- 当前范围不覆盖第三方未授权数据抓取。
- 当前范围不把 `group1` 与 `group2` 合并成一个混合大模型。
- 当前范围不保留 `group1` 新旧双方案长期并存。

## 风险与约束

- 若统一求解包持续停留在“设计里存在、发布里缺位”的状态，项目会继续被误解为训练工程。
- `group1` 的主要风险已从“类名是否足够”转为“query 切分、候选召回、向量混淆和全局分配是否足够稳定”。
- `group2` 若未来扩展为多缺口或复杂轨迹，当前业务语义和模型路线需要重评。
- Windows 驱动、CUDA 和 PyTorch 兼容性仍然是新手部署的主要风险。
- 若素材多样性不足，模型可能记住背景模板而不是目标特征。
