# 图形验证码专项训练手册

## 1. 这份文档适用于什么情况

这份文档不是给文字验证码用的。

它适用于下面这种验证码：

- 小图里给你一个或多个图形
- 大图里藏着对应的图形位置
- 你要找出大图中的正确位置

也就是：

- 点选图标验证码
- 图形匹配验证码
- 缺口图形定位验证码

## 2. 根据你给的样本，我看到的是什么

你给了两组图片。

## 2.1 对这 4 张具体样图的直接判断

为了避免理解跑偏，这里直接按你发来的 4 张图说结论。

### 图 1 + 图 2

- 图 1 是查询图
- 图 2 是场景图
- 图 1 里有多个深色小图标剪影
- 图 2 里在风景背景上叠加了多个同风格图标
- 用户要做的事，大概率是按图 1 给出的顺序，在图 2 中依次点击对应图标

这说明第一组不是文字识别，也不是普通单目标检测。

它是：

- 多目标定位
- 顺序点击
- 查询图到场景图的匹配

### 图 3 + 图 4

- 图 4 是查询图
- 图 3 是场景图
- 图 4 是一个房子形状的小图块
- 图 3 里有一个明显的白色房子轮廓
- 用户要做的事，大概率是在图 3 中找出房子位置

这说明第二组是：

- 单目标图形定位
- 小图到大图的形状匹配
- 输出一个点或一个框

## 2.2 这两组图为什么和 OCR 完全不是一回事

OCR 的结果是字符串，比如：

- `AB12`
- `7k9m`

你这两组图需要的结果不是字符串，而是坐标。

所以这两组图的正确任务定义是：

- 第一组：`[(x1, y1), (x2, y2), (x3, y3)]`
- 第二组：`(x, y)` 或 `[x1, y1, x2, y2]`

如果任务定义错了，后面训练、标注、验收全都会错。

### 第一组

第一组的特点是：

- 小图是黑色背景
- 小图里有多个深色小图标
- 大图是风景图
- 大图里叠加了多个图标

这类验证码的本质不是 OCR。

它更像：

- 给你一组目标图标
- 让你在大图中按顺序找到这些图标的位置

所以它的输出不是文字字符串，而是：

- 第 1 个图标的坐标
- 第 2 个图标的坐标
- 第 3 个图标的坐标

### 第二组

第二组的特点是：

- 小图是单个房子形状图块
- 大图里有一个白色房子轮廓
- 你要把小图匹配到大图中的目标位置

这也不是 OCR。

它更像：

- 单目标图形定位
- 单模板匹配
- 单缺口位置识别

它的输出通常是：

- 一个中心点坐标
- 或一个目标框

## 3. 为什么不能按 OCR 文字验证码训练

OCR 的输入输出是这样：

- 输入：一张图
- 输出：`AB12C`

你现在这两组不是这个问题。

你现在的问题是：

- 输入：小图 + 大图
- 输出：坐标

所以如果你拿这两组去训练 OCR，会出现两个根本问题：

1. 任务定义错了
2. 输出形式错了

一句话：

这两组应该做“目标定位/匹配模型”，不是“文字识别模型”。

## 4. 最推荐的总体方案

对于你现在这两组样本，我不建议训一个混合万能模型。

最稳的方案是：

1. 先做一个类型判断器
2. 再分别走两条求解链路

### 4.1 最简单的类型判断方式

其实你连模型都不一定需要。

因为这两类图长得差异很大。

你可以直接按规则判断：

- 如果小图里有多个独立图标，就走第一组流程
- 如果小图里只有一个图块，大图里只有一个白色轮廓目标，就走第二组流程

### 4.2 为什么建议拆成两个模型

因为这两类验证码的任务不同：

- 第一组：多目标、顺序匹配、输出多个点
- 第二组：单目标定位、输出一个点或一个框

如果硬塞进一个模型，会有这些问题：

- 标注格式不统一
- 输出格式不统一
- 样本分布混乱
- 训练会互相干扰

所以建议：

- 第一组一个专项模型
- 第二组一个专项模型

## 5. 第一组应该怎么做

## 5.1 第一组的任务定义

第一组要做的不是“识别图片里写了什么字”。

而是：

给一个查询图标列表，在大图中找到对应图标的位置，并且按照小图给出的顺序输出坐标。

你可以把第一组拆成两个子任务：

1. 在大图中找出所有候选图标
2. 判断小图中的每个图标，分别对应大图中的哪一个候选图标

## 5.2 第一组最稳的技术路线

我建议第一组用两阶段方案。

### 阶段 A：先做图标检测

目标：

在大图里先把所有图标框出来。

输出类似这样：

```json
[
  {"bbox": [x1, y1, x2, y2]},
  {"bbox": [x1, y1, x2, y2]},
  {"bbox": [x1, y1, x2, y2]}
]
```

这个阶段可以用：

- YOLO 检测模型
- RT-DETR
- 其他轻量检测模型

第一次做，直接用 YOLO 最省事。

### 阶段 B：做图标匹配

目标：

把“小图中的第 1 个图标”匹配到“大图中某个检测框”。

这个阶段有两种做法。

#### 做法 1：图标类别固定

如果你的图标种类是固定的，比如就几十种、上百种。

那你可以直接给每个图标分一个类别：

- `icon_phone`
- `icon_house`
- `icon_plane`
- `icon_car`

这样大图检测模型就可以直接检测出：

```json
[
  {"class": "icon_phone", "bbox": [...]},
  {"class": "icon_house", "bbox": [...]}
]
```

然后你只需要把小图中的目标图标也识别成类别，再去大图里找同类目标。

这种做法最简单，但前提是图标集合稳定。

#### 做法 2：图标种类不固定

如果图标经常换，不适合手工维护类别表。

那就不要做固定分类，而是做“特征匹配”。

做法是：

- 检测器只负责在大图里框出所有图标
- 再训练一个图标编码器
- 把小图图标和大图候选框都编码成向量
- 用相似度匹配

这叫：

- Siamese Network
- 度量学习
- 图像检索匹配

第一次落地，如果你的图标库比较固定，优先用“固定类别”方案。

## 5.3 第一组要收集什么样的样本

第一组你至少要收 4 种数据。

### A. 大图原图

就是风景背景图加图标后的最终图。

### B. 小图查询图

就是用户看到的“要找的图标列表”。

### C. 每个目标图标在大图中的坐标

例如：

```json
{
  "targets": [
    {"order": 1, "class": "icon_phone", "center": [31, 15], "bbox": [20, 8, 42, 24]},
    {"order": 2, "class": "icon_leaf", "center": [65, 18], "bbox": [55, 10, 75, 26]},
    {"order": 3, "class": "icon_boat", "center": [92, 20], "bbox": [80, 12, 104, 28]}
  ]
}
```

### D. 所有干扰图标的坐标

这一步很重要。

因为大图里可能不止有目标图标，还有干扰图标。

如果你不把干扰图标也标出来，检测模型会学不完整。

## 5.4 第一组样本如何获取

只讲合规方式。

### 最推荐的方式

从你自己的验证码生成服务里直接导出。

也就是说：

当你自己的系统生成第一组验证码时，同时记录：

- 背景图文件
- 查询图文件
- 目标图标顺序
- 每个目标图标在大图中的位置
- 所有干扰图标在大图中的位置

如果这些验证码是你自己的服务生成的，那么后台其实本来就知道图标放在了哪里。

你只是把这些信息保存下来而已。

### 你至少要保存哪些字段

```json
{
  "captcha_type": "group1_multi_icon_match",
  "query_image": "query/000001.png",
  "scene_image": "scene/000001.png",
  "targets": [
    {"order": 1, "class": "icon_a", "bbox": [x1, y1, x2, y2], "center": [cx, cy]},
    {"order": 2, "class": "icon_b", "bbox": [x1, y1, x2, y2], "center": [cx, cy]}
  ],
  "distractors": [
    {"class": "icon_x", "bbox": [x1, y1, x2, y2]}
  ]
}
```

## 5.5 第一组的标注方式

如果你有生成端坐标，就不需要人工标注。

如果没有生成端坐标，那就人工标。

第一组人工标注至少要做两件事：

1. 在大图中把所有图标框出来
2. 标明哪个框对应小图中的第几个目标

你可以用任何常见标注工具：

- labelme
- CVAT
- Label Studio
- 其他支持框标注的工具

## 5.6 第一组建议的数据量

第一次能跑起来的最低量：

- 训练集：5000 对
- 验证集：1000 对
- 测试集：1000 对

更实用的规模：

- 训练集：20000 到 50000 对
- 验证集：3000 到 5000 对
- 测试集：3000 到 5000 对

如果图标种类特别多，样本量要继续加。

## 5.7 第一组的训练方案

### 如果图标类别固定

建议：

1. 训练一个带类别的检测模型
2. 小图也用同样类别体系
3. 推理时按顺序取对应类别的目标坐标

这是第一选择。

### 如果图标类别不固定

建议：

1. 训练一个“图标检测模型”，类别只有一个：`icon`
2. 再训练一个“图标特征匹配模型”
3. 用相似度把小图目标和大图候选框对应起来

### 第一组推理输出格式建议

```json
{
  "type": "group1_multi_icon_match",
  "points": [
    {"order": 1, "x": 31, "y": 15},
    {"order": 2, "x": 65, "y": 18},
    {"order": 3, "x": 92, "y": 20}
  ]
}
```

## 6. 第二组应该怎么做

## 6.1 第二组的任务定义

第二组更简单。

它是：

- 一个小图块
- 一个大图目标轮廓
- 找到这个目标轮廓位置

所以它是单目标定位问题。

## 6.2 第二组最推荐的方案

第二组建议先别急着训练深度模型。

先做规则法基线。

原因是你给的样本里，大图目标轮廓非常明显，是白色房子形状。

这类任务很多时候用下面这些传统方法就能解决：

- 二值化
- 轮廓提取
- 模板匹配
- 形状匹配

### 第二组推荐顺序

1. 先做规则法
2. 如果规则法不稳定，再训练模型

## 6.3 第二组规则法怎么做

第一次做，可以按这个思路：

1. 读取大图
2. 找接近纯白的区域
3. 提取轮廓
4. 找出房子形状的候选轮廓
5. 取目标轮廓中心点

如果大图中的目标一直都是“白色轮廓 + 背景图”，这条路线成本最低。

## 6.4 第二组什么时候需要训练模型

如果出现下面这些情况，就开始训练模型：

- 白色轮廓不再稳定
- 背景干扰变强
- 轮廓边界不明显
- 小图块形状变多、背景更复杂
- 规则法误检严重

## 6.5 第二组训练什么模型

第二组你有两个靠谱选择。

### 方案 A：目标检测

直接把大图中的目标区域框出来。

输出：

- 一个 bbox
- 一个中心点

这个最简单。

### 方案 B：分割模型

如果你想更精确，就做分割。

输出：

- 一张 mask
- 然后再算中心点或轮廓框

如果只是完成验证码，第一次不需要分割，目标检测已经够用。

## 6.6 第二组要收集什么样的样本

### A. 小图块图片

就是那个单独的房子图块。

### B. 大图背景图片

就是带目标轮廓的大图。

### C. 目标位置

最少保存：

- `bbox`
- `center`

如果你做分割，还可以保存：

- `mask`

示例：

```json
{
  "captcha_type": "group2_single_shape_locate",
  "query_image": "query/000001.png",
  "scene_image": "scene/000001.png",
  "target": {
    "class": "house",
    "bbox": [120, 44, 156, 88],
    "center": [138, 66]
  }
}
```

## 6.7 第二组样本如何获取

还是优先从你自己的生成服务里导出。

如果这个房子轮廓是你自己画上去的，那么后台一定知道：

- 目标放在哪里
- 目标框多大
- 中心点坐标是什么

只要保存下来就行，不需要人工标注。

## 6.8 第二组建议的数据量

如果只做目标检测：

- 训练集：3000 到 10000 对
- 验证集：500 到 2000 对
- 测试集：500 到 2000 对

如果规则法已经很稳，其实你甚至可以先不训模型。

## 6.9 第二组推理输出格式建议

```json
{
  "type": "group2_single_shape_locate",
  "point": {"x": 138, "y": 66},
  "bbox": [120, 44, 156, 88]
}
```

## 6.10 第二组先做一个不训练也能跑的基线

你这张房子样图非常适合先做规则法。

第一次落地，直接按这个顺序试：

1. 读取大图
2. 转灰度
3. 提取高亮区域
4. 找白色或接近白色的连通区域
5. 提取轮廓
6. 用面积、宽高比、顶点数、形状相似度筛掉无关区域
7. 取最像房子的那个区域
8. 输出中心点

你可以把这个基线理解成：

- 不训练
- 不标很多数据
- 先判断这类题是不是根本不需要模型

如果这个基线已经能稳定命中，那就没必要一开始就上深度学习。

## 6.11 第二组规则法失败时再升级成模型

如果你发现下面这些问题，就升级：

- 白色目标不再稳定
- 背景也出现很多接近白色的区域
- 目标轮廓被裁切、模糊、遮挡
- 查询图不再总是房子，而是变成很多不同图形

这时再做：

- 单类别目标检测
- 或单类别分割

不要反过来，一开始就把简单问题做重。

- 第二组：1 个专项模型，或者直接 1 套规则法

## 9. 训练和验收时看什么指标

这两组不要再看 OCR 的“整串准确率”。

你要看定位指标。

### 第一组指标

- 单个目标点命中率
- 整组顺序全部命中率
- 平均中心点误差
- 错序率

### 第二组指标

- 单点命中率
- 框 IoU
- 中心点误差
- 平均推理时间

## 10. 这两组如何落地到当前项目

当前仓库的核心接口是 OCR。

这意味着：

- 不要把这两组任务硬塞进 `classification()`
- 不要把这两组任务当作字符识别

正确做法是：

在当前项目之外，或者在当前项目里新增独立模块。

例如新增：

- `group1_solver.py`
- `group2_solver.py`

以及独立接口：

- `/captcha/group1/solve`
- `/captcha/group2/solve`

## 11. 我给你的最终建议

针对你现在这两组图，最稳的路线是：

1. 明确承认这不是 OCR
2. 拆成两类任务分别处理
3. 第一组做“多图标检测 + 匹配”
4. 第二组先做规则法，再决定是否训练检测模型

## 12. 给你一份最小落地决策

如果你现在就要开工，不要纠结，直接按下面做：

### 第一组

- 如果图标种类固定：训练 1 个多类别检测模型

### 第一版目标

- 第一组先做到“顺序点击准确率可用”
- 第二组先做到“目标点基本稳定命中”

## 12.1 如果你要最快做出第一版，直接按这条路径走

如果你的目标不是做论文，而是尽快做出能跑的第一版，那么直接这样拆：

### 第一步

- 第二组先做规则法
- 因为最简单，最快出结果

### 第二步

- 第一组先做数据导出
- 把查询图、大图、目标顺序、目标坐标全部保存下来

### 第三步

- 第一组先上检测方案
- 如果图标类别固定，就做多类别检测
- 如果图标类别不固定，就做通用检测加匹配

### 第四步

- 只在第一版跑通后，再补自动评估和失败样本回灌

这条路径比“一开始就做统一模型”更稳，也更容易让一个完全没训过模型的人真正做出来。
